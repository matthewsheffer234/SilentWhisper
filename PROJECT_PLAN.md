# Silent Whisper Project Plan

## 1. Project Overview

Silent Whisper is an offline-first, workspace-based messaging platform for project teams. It combines Slack-style channels, direct messages, group conversations, deep threaded replies, server-side presence, immutable audit logging, and configurable local LLM-powered AI utilities.

The system must run fully on local or intranet infrastructure. It must not depend on external CDNs, externally hosted assets, or public AI APIs.

Silent Whisper must also be deployable on the same server as the existing Silent Lattice stack described in `/root/wireservice/PROJECT_CONTEXT.md`. It should be able to run as an independently built app while being served through the same shared nginx entrypoint (`wireservice-nginx-1`), on its own dedicated hostname: `whisper.silentlattice.dev`.

**Target scale**: up to 100 concurrent users on a single backend instance. This number is a load-bearing design constraint, not a rough guess — see Scalability Target below. It determines whether in-process state (presence, rate limiting) is acceptable or whether a shared store is required.

## 2. Target Architecture

### Monorepo Layout

```text
/frontend   Vite + React client application
/backend    Node.js API, WebSocket server, auth, audit, and configurable LLM proxy
/database   PostgreSQL schema, migrations, grants, and seed data
/scripts    audit verification and local maintenance utilities
```

### Core Services

- PostgreSQL stores users, workspaces, channels, messages, memberships, refresh tokens, and audit logs.
- Node.js exposes REST APIs for auth, workspace/channel management, message pagination, admin audit views, and AI actions.
- WebSockets handle real-time messages, thread updates, room membership state, and presence heartbeats.
- React renders the 3-column messaging interface and admin tooling.
- Local LLM services are accessed only through a backend AI provider proxy.

### Scalability Target

- The system must support up to 100 concurrent active users (concurrent WebSocket connections plus concurrent REST traffic) on a **single** backend instance. Design for this explicitly rather than leaving it implicit.
- Presence and room-membership state may live in backend process memory because the deployment target is a single Node.js instance. Do not introduce a multi-instance requirement (PM2 cluster mode, horizontal replicas) without also introducing a shared state store (e.g., Redis pub/sub) for presence and WebSocket fan-out — a single instance with in-memory state and a multi-instance deployment without shared state are the two failure modes to avoid.
- Size the PostgreSQL connection pool (`pg` Pool `max`) to comfortably serve 100 concurrent users without exhausting `PGDATABASE`'s `max_connections`, and leave headroom for the existing Silent Lattice services (Oracle, Elasticsearch, Ollama, API/frontend containers) sharing the same host. Default to `max: 20` (up to `30` if profiling shows queueing) rather than sizing the pool to the user count 1:1 — Node's async event loop multiplexes many concurrent browser connections through a small number of pooled DB connections without saturating Postgres, and an oversized pool just burns Postgres memory/backend processes for no throughput gain at this scale.
- Add covering indexes for every authorization-check query path (membership lookups by `user_id`) so per-request authorization does not degrade as usage grows — see PostgreSQL Schema.
- Paginate all message history queries server-side; never return unbounded result sets regardless of channel size.
- Load-test the WebSocket and REST layers at 100 simulated concurrent users before Phase 5 is considered complete, and record observed p95 message-delivery latency and API response times as a baseline for future capacity planning.
- If usage is expected to exceed 100 concurrent users later, treat that as a distinct scaling milestone (shared presence store, sticky-session load balancing, connection-pool retuning) rather than assuming the current design silently scales further.

### Serving Under Silent Lattice

The current main server uses a Docker Compose stack behind the persistent `wireservice-nginx-1` reverse proxy, defined in `/root/wireservice/nginx/nginx.conf`. That file currently has two `server_name` blocks — `silentlattice.dev`/`www.silentlattice.dev` and `dev.silentlattice.dev` — each with its own certbot cert directory under `/etc/letsencrypt/live/`. Silent Whisper is served as a **third, dedicated `server_name` block** for `whisper.silentlattice.dev`, not as a path prefix under an existing domain. This means Silent Whisper owns `/`, `/api`, `/ws`, and `/health` outright on its own hostname — it does not need to coexist with Silent Lattice's routes on the same origin.

- Do not develop directly in `~/wireservice`; use the appropriate development repo or the Silent Whisper repo, then promote deliberately.
- **Network integration pattern** (updated 2026-07-12 after actually deploying this — see Section 11's "Production Deployment" entry for the full story): this host runs two patterns side by side — prod (`~/wireservice`) joins `wireservice_default` directly and nginx proxies via service name (`http://frontend:3000`); dev (`~/wireservice-dev`) reaches its equivalent ports via `http://host.docker.internal:3001`/`:8001`, which works there only because those ports are bound to `0.0.0.0`. Silent Whisper's ports are deliberately bound to `127.0.0.1` instead (see below) — and `host.docker.internal` traffic arrives via the Docker bridge gateway, a different source address than localhost, which a loopback-only bind rejects. So Silent Whisper actually uses a **third pattern**: its own independent Compose stack (`postgres`/`migrate` stay isolated), but `backend` and `frontend` are *also* attached to `wireservice_default` (declared as an `external` network in `docker-compose.yml`), and nginx addresses them by container name (`silentwhisper-backend-1:8000`, `silentwhisper-frontend-1:3000`) — the same mechanism prod uses, not dev's. This was originally planned as the dev/`host.docker.internal` pattern; that assumption turned out to be wrong the first time it was actually deployed, and is corrected here rather than left stale.
- Publish Silent Whisper's frontend/backend ports bound to `127.0.0.1` only (e.g. `127.0.0.1:3101:3000`), not `0.0.0.0`. (Note: `wireservice-dev-frontend-1`/`-api-1` currently publish on `0.0.0.0`, which is directly reachable from the internet bypassing nginx/TLS entirely — don't repeat that; bind loopback-only like the Oracle/Elasticsearch containers already do.)
- Avoid host port collisions with Silent Lattice services. The existing stack already uses `3000`/`3001`/`8000`/`8001` for frontend/API, plus Oracle, Elasticsearch, and Ollama ports.
- DNS: **done** — `whisper.silentlattice.dev` already resolved when checked (2026-07-12), implying a wildcard record already exists at the registrar. No GoDaddy change was actually needed.
- TLS: **done**, via `certbot certonly --standalone -d whisper.silentlattice.dev` (2026-07-12) — the user explicitly chose to match the existing two domains' method rather than migrate to `webroot` as originally recommended here. That recommendation still stands as the better long-term fix (see the certbot renewal-hooks problem discovered in Section 11 — right now **none of the three domains** can actually auto-renew), just not the one taken this round.
- Nginx: **done**. Not `host.docker.internal` as originally planned here — see the corrected Network Integration Pattern note above. The actual working blocks proxy `/` to `silentwhisper-frontend-1:3000`, `/api/` to `silentwhisper-backend-1:8000` (no rewrite — Silent Whisper's Express app already expects the `/api` prefix itself, unlike the other two blocks' APIs), `/health` the same, and `/ws` with the upgrade headers below.
- WebSocket proxying: **done** — `proxy_http_version 1.1;`, `proxy_set_header Upgrade $http_upgrade;`, `proxy_set_header Connection "upgrade";` on the `/ws` location, verified with a real WebSocket client completing an `authenticate` round trip through nginx, not just assumed from the config.
- `sl-admin.py`'s nginx-provisioning wizard (`_write_nginx_conf`) is hardcoded to exactly two domains (prod + dev) — confirmed still true, and moot besides: its rebuild/restart/reload menu options assume `wireservice-nginx-1` is a `docker compose` service, which it is not (confirmed — no `nginx` service exists in `/root/wireservice/docker-compose.yml`). The config was hand-edited directly, and the container rebuilt/reloaded manually (see RUNBOOK.md, Production Deployment).
- Because Silent Whisper owns its whole origin, no path-prefix or base-path configurability (`/silent-whisper/...`) is needed anywhere in the app. `VITE_API_URL` points at `https://whisper.silentlattice.dev/api` and `VITE_WS_URL` at `wss://whisper.silentlattice.dev/ws`, the same pattern Silent Lattice already uses for its own `VITE_API_URL`.
- Provide a plain `/health` backend endpoint — no path-prefix collision risk since it's on its own hostname.

### Configurable LLM Provider Settings

The AI integration must be configurable after initial deployment. Do not hard-code a single vLLM URL, model name, prompt format, or timeout in application code.

Treat `vllm`, `ollama`, and `disabled` as three equally first-class values for `LLM_PROVIDER`, not "vLLM now, maybe Ollama someday." This test environment has no GPU; the target production network this eventually moves to has vLLM backed by dedicated GPU hardware. Moving between them must be a config change only:

- Define one provider-adapter interface (send prompt, receive completion, report health) that both the vLLM adapter and the Ollama adapter implement identically. AI proxy code above that line must never branch on which provider is active except inside the adapter factory that picks one from `LLM_PROVIDER`.
- **This host/test environment**: default to `LLM_PROVIDER=ollama`, running a dedicated Ollama container for Silent Whisper (isolated from Silent Lattice's own two Ollama instances, so restarting/upgrading one doesn't affect the other). This host has no GPU and 8 CPU cores; Silent Lattice's existing Ollama containers already prove CPU-only Ollama inference works fine here for occasional use — vLLM's CPU backend is comparatively unproven and likely to underperform without a GPU.
- **Target production network**: `LLM_PROVIDER=vllm` with `LLM_BASE_URL` pointed at the GPU-backed vLLM endpoint. No code change required — only environment/config.
- On startup and periodically thereafter, the backend health-checks the configured provider and surfaces reachability (up/down, last check time) through the admin settings surface, so a misconfigured or unreachable provider is visibly diagnosable rather than a silent failure the first time someone clicks "Summarize."
- Bound the number of concurrent in-flight LLM requests via `LLM_MAX_CONCURRENT_REQUESTS`, enforced by the AI proxy regardless of provider. This is a hardware-capacity control, not just an abuse control (see Rate Limiting & Abuse Prevention): a CPU-only single Ollama instance can realistically serve close to one generation at a time before requests queue up for tens of seconds each, so default this low (e.g., `1`–`2`) for the Ollama-backed test environment and raise it once running against GPU-backed vLLM, which can sustain meaningfully more concurrency.

Backend configuration must support:

```text
LLM_PROVIDER                  # ollama for this test environment; vllm once moved to the GPU-backed network; disabled to turn AI features off entirely
LLM_BASE_URL                  # e.g. http://silent-whisper-ollama:11434 here, or http://vllm:8000 / http://host.docker.internal:8002 on the target network
LLM_MODEL                     # model identifier sent to the provider (e.g. mistral to match Silent Lattice's existing Ollama models)
LLM_API_KEY                   # optional secret for protected local gateways
LLM_TIMEOUT_MS
LLM_MAX_INPUT_CHARS
LLM_MAX_OUTPUT_TOKENS
LLM_MAX_CONCURRENT_REQUESTS   # caps in-flight LLM proxy calls; low default for CPU-bound Ollama, higher once on GPU-backed vLLM
LLM_TEMPERATURE
LLM_STREAMING_ENABLED
LLM_SUMMARY_PROMPT_VERSION
LLM_TASK_PROMPT_VERSION
```

Provide an admin-only settings surface or config file workflow that can update non-secret LLM options later without code changes. Secrets must remain in environment variables or a local secret store, not in frontend code or committed files.

## 3. Security Baseline

Security is designed in from Phase 1, not bolted on during hardening in Phase 5. The items below are minimums given Silent Whisper handles authentication, private conversations, and an audit trail — treat every one as a Phase 1/2/3 requirement, not a Phase 5 nice-to-have. The Phased Implementation Roadmap below places each item in the phase where it must first be implemented.

### Authentication & Session Security

- Hash passwords with `bcryptjs` using a work factor of at least 12; do not lower this for performance without re-evaluating the threat model. Always call the async API (`bcrypt.hash`/`bcrypt.compare`, not a sync variant) — a work-factor-12 hash blocks the event loop for ~200-300ms, and that same event loop is broadcasting to every connected WebSocket client. Async bcrypt runs on libuv's threadpool instead; be aware it shares that threadpool with other Node internals, so a burst of concurrent signups can still cause minor contention — not a concern at this scale, but don't add more threadpool-heavy work without checking `UV_THREADPOOL_SIZE`.
- Enforce a minimum password length and reject a small deny-list of common/breached passwords at signup, rather than relying on complexity rules alone.
- Issue short-lived JWT access tokens (e.g., 15 minutes) plus a longer-lived refresh token. Store refresh tokens hashed in a `refresh_tokens` table (see PostgreSQL Schema) so they can be revoked on logout, admin action, or detected compromise.
- Store the access token in memory (JS variable/React state) and the refresh token in an `httpOnly`, `Secure`, `SameSite=Strict` cookie scoped to the app's base path. Do not store tokens in `localStorage` or `sessionStorage` — either turns a single XSS bug into full account takeover.
- Rate-limit login and signup endpoints per IP and per username to slow credential-stuffing and brute-force attempts. An in-process token-bucket limiter is sufficient at the single-instance, 100-user scale targeted here (see Scalability Target); a shared store (Redis) would only be required if the backend ever runs multiple instances.
- Every failed login attempt is an audit event (see Forensic Security Audit Log). The raw events must exist from Phase 2 even before any alerting is layered on top.

### WebSocket Authentication Handshake

- The browser `WebSocket` constructor cannot send custom headers, so the in-memory access token (which never touches `localStorage`) cannot ride an `Authorization` header on the initial handshake the way REST calls do. Resolve this explicitly rather than improvising per-feature:
  - **Preferred**: open the socket unauthenticated, then immediately emit a client→server `authenticate` event frame carrying the access token as the first message on the open connection. The server holds the connection in an unauthenticated state — no room joins, no data streamed — until that frame validates, then treats the connection as identified for its lifetime.
  - **Acceptable fallback**: pass the token as a handshake query parameter (`/ws?token=...`) only if the `authenticate`-frame approach is impractical for the chosen WebSocket library. If used, ensure nginx and any backend request logging redact the `token` query parameter, since query strings are otherwise logged in plaintext and can leak via `Referer` headers — this is strictly worse for token exposure than the frame-based approach, so treat it as a fallback, not a default.
  - Either way, re-run the same authentication step on reconnect; a reconnecting socket is unauthenticated until it proves otherwise again.
- **Long-lived connections outlive the access token.** A chat WebSocket routinely stays open for hours, but the access token used to authenticate it expires in ~15 minutes (Authentication & Session Security, above). Resolve this explicitly rather than leaving the socket trusting an expired credential indefinitely:
  - The client re-sends a fresh `authenticate` frame with a renewed access token (obtained silently via the refresh-token flow) on a timer well before the current token expires (e.g., every 10 minutes for a 15-minute token).
  - The server tracks each connection's token expiry and disconnects (forcing a reconnect, which re-authenticates) if no renewed `authenticate` frame arrives before it. This keeps the "short-lived access token" property meaningful for WebSocket sessions, not just REST calls.

### Audit Log Write Serialization

- The `prev_row_hash`/`curr_row_hash` chain (Section 4) requires that row N's write read row N-1's hash. At 100 concurrent users, two events (e.g., two failed logins, or two AI audit events) can fire in the same millisecond; if two concurrent writers both read the same "latest row" and then both insert, the chain forks and verification breaks.
- Serialize the read-latest-hash-then-insert step with a Postgres advisory lock (`pg_advisory_xact_lock` on a fixed, well-known key, held for the duration of the transaction that reads the last row and inserts the new one). This is the primary correctness guarantee — it holds regardless of how many backend processes are running, which matters because an in-process-only mutex silently breaks the instant a rolling/blue-green deploy briefly runs an old and new container at the same time. An in-process queue in front of the advisory lock is fine as an optimization (avoids most writers ever contending on the lock) but must not be the only guard.
- Do not rely on "insert and hope" or on database default isolation to prevent this. The advisory lock approach means the audit write path staying single-instance (Scalability Target) is no longer a correctness requirement for the audit chain specifically — it remains a requirement for presence state and rate limiting, which are not lock-protected the same way.

### Authorization Model

- Authorization is enforced server-side on every REST call and every WebSocket event — never inferred from what the UI happens to render. A user who is not a member of a channel must get a 403 (REST) or a closed connection (WebSocket) even if they know its UUID.
- Centralize membership checks (workspace membership, channel membership) in one shared backend module used by both REST handlers and WebSocket event handlers, so the rule is written once and cannot drift between the two transports.
- Private channels, direct messages, and group DMs must never be joinable, listable, or readable by non-members, including via search or admin tooling. The admin audit dashboard is the one intentional exception, and that exception is itself an audited action.
- WebSocket room joins re-validate membership at join time and again on every reconnect — do not trust a previously-granted room membership across reconnects.

### Input Handling & Injection Prevention

- Treat all message content, workspace names, and channel names as untrusted. Use parameterized queries everywhere via Knex.js (Phase 1), which parameterizes by construction; never build SQL with raw string interpolation, including inside `knex.raw()` calls (use its bound-parameter form there too).
- The frontend escapes/sanitizes message content before rendering. React's default JSX escaping covers plain text; if Markdown or rich-text rendering is added later, sanitize with an allow-list HTML sanitizer rather than passing raw content to `dangerouslySetInnerHTML`.
- Enforce a maximum message length server-side (not just in the UI) to bound storage, audit payload size, and LLM prompt size.
- Validate all UUIDs, enum-like fields (`channels.type`, `system_role`), and pagination parameters server-side; reject malformed input with 400s rather than passing it through to the database.

### Transport & Headers

- All traffic terminates TLS at the shared `wireservice-nginx-1` proxy; the backend does not implement TLS itself, but must correctly honor `X-Forwarded-Proto`/`X-Forwarded-For` when computing audit `actor_ip` values and any absolute URLs.
- Set standard security headers at the app layer (or confirm the `whisper.silentlattice.dev` nginx block sets them): a `Content-Security-Policy` with no `unsafe-inline` scripts and no third-party origins (consistent with the offline-only requirement), `X-Content-Type-Options: nosniff`, and `Referrer-Policy: same-origin`.
- Configure CORS narrowly to `https://whisper.silentlattice.dev` only (plus a local dev origin such as `http://localhost:5173` in non-prod config) — never a wildcard origin, especially once cookies carry credentials. Since Silent Whisper owns its whole subdomain, this is a single fixed origin, not a set of path-prefixed exceptions.

### Rate Limiting & Abuse Prevention

- Rate-limit message sends per user/connection so a single client cannot flood a channel or overwhelm WebSocket broadcast fan-out.
- Rate-limit AI proxy calls (summarize, extract tasks) per user. Local LLM inference is comparatively expensive, and an unbounded loop from a buggy or malicious client could starve the shared provider for every user on the host.
- Enforce `LLM_MAX_CONCURRENT_REQUESTS` as a global cap on in-flight AI proxy calls, independent of the per-user rate limit above — on the CPU-only Ollama-backed test environment, total inference throughput (not just any single user's request rate) is the actual bottleneck.
- Cap concurrent WebSocket connections per user (a small fixed number of tabs/devices) to bound per-user resource use.

### LLM-Specific Risks

- Message content sent into summarization or task-extraction prompts is untrusted user input. Delimit it clearly from system/instruction text in the prompt template (fenced or tagged blocks) so injected instructions inside a message ("ignore previous instructions and...") are treated as data, not as commands to the model.
- Truncate input to `LLM_MAX_INPUT_CHARS` server-side before prompt construction, not just as a UI hint.
- Treat model output as untrusted when rendering it in the UI — apply the same escaping rules as message content, since a prompt-injected response could otherwise attempt stored XSS via the summary text.
- Log the prompt template version and truncated input length (not full content) in the AI audit event so injection attempts remain traceable after the fact.

### Secrets & Configuration

- Zero hardcoding: no API key, password, token, database credential, or secret value is ever written into application code, tests, config committed to the repo, or documentation — including example snippets in `CLAUDE.md`/runbooks, which must use placeholders.
- `JWT_SECRET`, `LLM_API_KEY`, and `PGPASSWORD` live only in environment variables or a local secret store — never in `app_settings`, frontend code, or committed files.
- Load all environment-varying configuration (all of Local Configuration and Configurable LLM Provider Settings, above) dynamically via environment variables, using `dotenv` for local development in both `/frontend` (Vite's built-in `.env` handling) and `/backend`.
- Commit a `.env.example` at the root of `/backend` and `/frontend` with every variable name the app reads and a placeholder value (e.g., `JWT_SECRET=your_jwt_secret_here`, `PGPASSWORD=your_db_password_here`) — never a real value. This is the only environment-variable file ever committed; real `.env`/`.env.local` files are created locally by whoever runs the app and are never checked in.
- A `.gitignore` excluding `.env`, `.env.local`, and any cloud-credential directories (e.g. `.aws/`) must exist before the first commit to the Silent Whisper repo — create it in the same commit that initializes the monorepo structure (Phase 1), not after code already exists.
- If CI/CD (GitHub Actions) or infrastructure-as-code is added later, reference secrets only as repository secrets (`secrets.*` in Actions) or cloud secret-manager identifiers — never embed raw values in workflow YAML or IaC templates.
- Design JWT verification so a `JWT_SECRET` rotation invalidates outstanding tokens predictably (e.g., a key ID embedded in the token header) rather than requiring a hard cutover that silently logs everyone out.

### Dependency Hygiene

- Run `npm audit` (or equivalent) for both `/frontend` and `/backend` before each commit that changes dependencies. Do not add a dependency with a known critical vulnerability without a documented reason.

## 4. PostgreSQL Schema

### Users And Security

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE TABLE workspace_members (
    workspace_id UUID NOT NULL,
    user_id UUID NOT NULL,
    system_role VARCHAR(20) NOT NULL DEFAULT 'MEMBER',
    PRIMARY KEY (workspace_id, user_id)
);

CREATE INDEX idx_workspace_members_user ON workspace_members(user_id);

CREATE TABLE refresh_tokens (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash VARCHAR(255) NOT NULL,
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    revoked_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_refresh_tokens_user ON refresh_tokens(user_id);
```

`system_role` supports:

- `ADMIN`
- `MEMBER`

`refresh_tokens` stores only a hash of the token (never the raw value), so revocation (logout, admin action, detected compromise) is a row update, not a secret-recovery risk.

### Layout And Structural Hierarchy

```sql
CREATE TABLE workspaces (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name VARCHAR(100) NOT NULL,
    owner_id UUID REFERENCES users(id),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE TABLE channels (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    workspace_id UUID REFERENCES workspaces(id) ON DELETE CASCADE,
    name VARCHAR(100) NOT NULL,
    type VARCHAR(20) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE TABLE channel_members (
    channel_id UUID REFERENCES channels(id) ON DELETE CASCADE,
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    PRIMARY KEY (channel_id, user_id)
);

CREATE INDEX idx_channel_members_user ON channel_members(user_id);
```

`channels.type` supports:

- `PUBLIC`
- `PRIVATE`
- `DIRECT`
- `GROUP_DM`

The `idx_channel_members_user` and `idx_workspace_members_user` indexes exist specifically to keep per-request authorization checks (Section 3) fast as message volume and membership counts grow toward the 100-concurrent-user target.

### Communication And Content

```sql
CREATE TABLE messages (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    channel_id UUID REFERENCES channels(id) ON DELETE CASCADE,
    user_id UUID REFERENCES users(id),
    content TEXT NOT NULL,
    parent_message_id UUID REFERENCES messages(id) ON DELETE CASCADE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_messages_channel_date ON messages(channel_id, created_at DESC);
CREATE INDEX idx_messages_threading ON messages(parent_message_id)
    WHERE parent_message_id IS NOT NULL;
```

Messages with `parent_message_id IS NULL` belong to the main channel feed. Messages with `parent_message_id` set belong to a thread attached to the referenced message. Enforce the maximum message length (Section 3) at the API layer before insert.

### Forensic Security Audit Log

```sql
CREATE TABLE audit_logs (
    id BIGSERIAL PRIMARY KEY,
    timestamp TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    actor_id UUID NOT NULL,
    actor_ip VARCHAR(45) NOT NULL,
    action_type VARCHAR(100) NOT NULL,
    target_resource VARCHAR(255),
    payload JSONB,
    prev_row_hash VARCHAR(64) NOT NULL,
    curr_row_hash VARCHAR(64) NOT NULL
);

CREATE INDEX idx_audit_logs_timestamp ON audit_logs(timestamp DESC);
CREATE INDEX idx_audit_logs_actor ON audit_logs(actor_id);
```

`audit_logs` is append-only. Every inserted row must compute `curr_row_hash` with SHA-256 using the row's canonical audit payload plus the previous row's `curr_row_hash`.

Tracked audit events include:

- Authentication logins
- Authentication failures
- User role changes
- Workspace access changes
- Channel membership changes
- AI summarization requests
- AI action item extraction requests
- Data export requests
- Admin audit verification attempts
- Admin access to private channel/DM content via the audit dashboard exemption (Section 3)

Store non-secret application settings in the database so operational values can change after deployment without code edits or frontend rebuilds.

```sql
CREATE TABLE app_settings (
    key VARCHAR(100) PRIMARY KEY,

```

## 9. Rules Of Engagement For AI Agents

- Do not write code that fetches assets over the internet.
- Do not use external CDNs.
- Do not rely on browser time for trusted timestamps.
- Use server-generated timestamps for messages, presence, and audit records.

- Write tests alongside each backend module, frontend feature, and script, including negative authorization tests (non-member denied).

- Keep LLM provider access server-side only.
- Do not hard-code deployment paths, public URLs, LLM endpoints, model names, or host ports.

- Authentication, message-send, and AI proxy endpoints are rate-limited per user/IP.
- A load test confirms the system serves 100 concurrent simulated users with documented p95 message-delivery and API latency.

### Bounded AI concurrency queue instead of hard-rejecting (2026-07-18)

Implemented `FEATURE_REQUEST.md` entry 2 (`ARCHITECTURE_REVIEW_(Claude).md` P1 Recommendation 6): a second person clicking Summarize/Extract Tasks/Catch Me Up while one AI request is already in flight used to get an immediate `503`, not a queue position — real at `LLM_MAX_CONCURRENT_REQUESTS=1`, this environment's default.

- **`backend/src/llm/concurrencyGate.js`**: `tryAcquire()`/`release()` became async `acquireSlot(maxConcurrent, { onQueued })`/`release()`, backed by a bounded in-memory FIFO array of pending resolvers (new `config.llm.queueMaxDepth`, env `AI_QUEUE_MAX_DEPTH`, default 8, operational-only like `digestMaxWindowHours`). A request under the concurrency cap still grants a slot immediately (`Promise.resolve()`, `onQueued` never fires); one over the cap but under the queue-depth limit gets queued, `onQueued` firing synchronously (inside the `Promise` executor, before `acquireSlot` even returns) with its 1-based position; one arriving when the queue is already at `queueMaxDepth` gets `Promise.reject(...)` immediately — narrowing *when* the capacity rejection fires, not removing it. `release()` now checks the queue first: if anything is waiting, the freed slot transfers directly to the longest-waiting entry (`queue.shift()` then resolve) rather than decrementing `inFlight` and letting a brand-new caller race a queued one for the same slot.
- **`backend/src/llm/aiService.js`'s `runStreamingCompletion`**: reorders prompt construction (`promptBuilder(...)`) to run *before* `acquireSlot`, a deliberate departure from the design text's literal ordering — prompt construction is pure, local truncation/formatting with no provider call, so it needs no concurrency slot, and doing it first means every response header (including the new `X-Ai-Queue-Position`) is computed and available in one place before any header gets flushed. This mattered because `onQueued`'s callback calls `res.flushHeaders()` immediately (to give the client a real signal while still queued) — `res.setHeader()` after an early flush throws, so the queued and not-queued paths share one `setCompletionHeaders()` closure, called either inside `onQueued` (queued case) or once `acquireSlot` resolves (immediate-grant case), never both. The queue-full rejection is caught and re-thrown as the existing `ServiceUnavailableError('AI service is at capacity, please try again shortly')` — same message every caller/test already expected. No change needed to the three call sites (summarize/extract-tasks/workspace-digest routes) — all three already share this one function.
- **Frontend**: `frontend/src/api/ai.js`'s `streamPost` reads the `X-Ai-Queue-Position` response header (available the instant `fetch()`'s promise resolves — well before the streamed body finishes, since headers arrive first) and calls a new `onQueued` option if present, threaded through `summarizeChannel`/`extractTasks`/`requestWorkspaceDigest`. `frontend/src/aiPresentation.js` gained `formatAiQueueLabel(position)` ("Queued (position N)…"). `ChannelView.jsx`/`ThreadSidebar.jsx`/`WorkspaceDigestPanel.jsx` each track a `queuePosition` field in their summary/tasks/digest state, showing the queued label on their AI Actions trigger/submit button while waiting and clearing it back to "Running AI…"/"Generating…" on the first streamed chunk (a chunk arriving means a slot was actually granted).
- **Tests**: `backend/tests/llmConcurrencyGate.test.js` rewritten end to end for the async API — immediate grant with no queuing, a single queued request's FIFO position and eventual grant on `release()`, three queued requests granted strictly in FIFO order as slots free up one at a time, rejection once queue depth is exceeded (queue depth itself unaffected by the rejected arrival), and `release()` with nothing queued still decrementing/never going negative. `backend/tests/aiRoutes.test.js` gained a new `AI concurrency queue` describe block exercising the real HTTP routes (not the gate directly): a second request queues and completes once the first finishes; a third and fourth queue and are granted strictly in FIFO order; an arrival at full queue capacity gets an immediate `503` and audits nothing. Two real bugs surfaced building these, both fixed in the test file rather than worked around: (1) supertest/superagent's `Test` object is lazy — merely constructing `request(app).post(...).send(...)` without immediately awaiting/`.then()`-ing it never sends the request over the wire, so a new `fireNow()` helper wraps an explicit `.end()` call in a plain `Promise` to actually get two requests genuinely in flight at once, a prerequisite for testing queuing behavior at all; (2) draining several requests' mocked `fetch` calls at test end must happen one resolver at a time, polling for each to actually be reached before triggering it — a single `pendingResolvers.forEach(resolve => resolve())` only resolves whichever fetch calls existed at that one instant (just the currently in-flight request's), permanently hanging every request still waiting in the queue for its own turn to reach `fetch`.
- **Docs**: `.env.example` (root and `backend/`) and `docker-compose.yml` gained `AI_QUEUE_MAX_DEPTH`; `RUNBOOK.md`'s AI Features env-var table, streaming-response-format section, and the `503 "AI service is at capacity"` common-problem entry updated to describe the queue and the new header.

### Hot path splitting: async notification writes and entity linking (2026-07-18)

Implemented from `FEATURE_REQUEST.md`'s "hot path splitting" entry, the last remaining item from the two architecture reviews' backlog. The entry's own design text flagged a real UX tradeoff needing explicit sign-off rather than an assumed default — mention notifications would go from same-request-instant to lagging behind message delivery by roughly one poll interval — and offered a fallback (notifications stay synchronous, only entity linking moves) alongside the full move. Same `FOR UPDATE SKIP LOCKED`-via-CTE claim pattern as `embedding_jobs` (avoiding the same over-claim trap that table's own migration comment documents), same grant set.
- **Enqueue (`backend/src/services/messageSideEffectsQueue.js`)**: `enqueueMessageSideEffectJobs(db, { messageId, workspaceId })` — always inserts a `NOTIFICATION` row (`mention_notifications.workspace_id` is nullable, so DMs get mention notifications too, matching the pre-existing behavior it replaces); inserts an `ENTITY_LINK` row only when `workspaceId` is truthy, since DIRECT/GROUP_DM channels have a `NULL` `workspace_id` and entities are workspace-scoped. Failure-tolerant by the same contract as `enqueueEmbeddingJob`: a queue-insert failure is logged, never propagated to fail the message-send response.
- **Send-path changes (`backend/src/routes/messages.js`, `backend/src/ws/server.js`)**: both `POST /channels/:channelId/messages` and `handleMessage` replaced their inline `extractMentionedUserIds`/`createMentionNotifications`/`linkMessageEntities` calls and the per-mention `sendToUser({ type: 'mention', ... })` push with one `enqueueMessageSideEffectJobs` call, right after the existing `broadcastToRoom`. - **Worker (`backend/src/workers/messageSideEffectsWorker.js`)**: structurally identical to `embeddingWorker.js` — `running` guard against overlapping ticks, `setInterval` sweep (new `config.messageSideEffects.workerIntervalMs`, default 1s — tighter than embedding's 2s default, since these jobs feed a user-visible "you were mentioned" notification rather than background search indexing, and there's no external provider to throttle against). ### WebSocket connection hygiene: immediate eviction on account disable + payload cap (2026-07-17)

Implemented from `FEATURE_REQUEST.md` entry 1, the top P0 item independently flagged by both `ARCHITECTURE_REVIEW_(Claude).md` and `ARCHITECTURE_REVIEW_(CODEX).md`: a disabled account kept live REST and WebSocket access until its access token naturally expired, and the WebSocket gateway accepted frames of unbounded size.

- **Immediate eviction (`backend/src/ws/connectionRegistry.js`, `backend/src/routes/admin.js`)**: new `disconnectUser(userId, { code = 4004, reason = 'Account disabled' })` iterates a user's currently-registered sockets, sends an explicit `{ type: 'error', error: reason }` frame (a close `reason` string isn't reliably surfaced to browser client code the way a message frame is), then closes each socket — `ws/server.js`'s existing `close` handler does the usual `unregisterConnection`/`leaveAllRooms`/`handleDisconnect` cleanup, none of it duplicated here. `POST /api/admin/users/:userId/disable` calls it synchronously, immediately after `revokeAllRefreshTokensForUser`, in the same request that flips `status` to `DISABLED` — not queued, since "immediately" was the point of this entry.
- **Status re-check on every identity-establishing point (`backend/src/auth/requireAuth.js`, `backend/src/ws/server.js`)**: eviction alone only closes sockets that are open *right now* — it doesn't stop a still-unexpired access token from opening a new connection or making a new REST call a moment later. `requireAuth` now queries `users.status` after JWT verification on every REST request (previously trusted the token claims alone) and rejects a non-`ACTIVE` user with the identical "Invalid or expired access token" message an actually-invalid token gets — no distinct message that would let a disabled account tell the two apart, mirroring how `routes/auth.js`'s login rejection already treats a disabled account the same as a wrong password. `ws/server.js`'s `handleAuthenticate` gained the identical check, running before the first-auth/reauth branch so it covers both a brand-new connection and a renewal frame on an already-open one.
- **`maxPayload` (`backend/src/config.js`, `backend/src/ws/server.js`)**: new `WS_MAX_PAYLOAD_BYTES` (default `131072`, 128 KiB — sized above the worst-case UTF-8 encoding of a `MAX_MESSAGE_LENGTH` (10,000-char) message plus JSON framing overhead, so no legitimate message frame is ever at risk) passed to `new WebSocketServer({ maxPayload })`. The `ws` library rejects and closes an over-limit frame before it's ever buffered or reaches this app's `JSON.parse`, closing an unauthenticated memory/CPU exhaustion vector.
- **A real bug found by testing, not by inspection**: exceeding `maxPayload` surfaces as an `'error'` event on the per-connection socket — `ws`'s receiver rejects the frame before `'message'` ever fires — and `wss.on('connection', ...)` registered no `'error'` listener on individual sockets anywhere in the file. An EventEmitter `'error'` event with no listener is a fatal, process-crashing exception in Node, not merely a dropped event; the very first test run of the new `maxPayload` test crashed the backend test server. Fixed with a no-op `ws.on('error', () => {})` per connection — `ws` already closes the socket itself (code `1009`) once it emits this, so no further handling was needed.
- **Docs**: `RUNBOOK.md`'s WebSocket Protocol close-code line gained `4004` (account disabled while connected) and a note on the new payload cap closing with code `1009`.
- **Tests**: `backend/tests/ws.test.js` — disabling a connected user force-closes their socket within the same test, not just on a later reconnect; a disabled user's still-unexpired token is rejected on a fresh authenticate attempt; a disabled user's still-unexpired token is rejected on a re-authenticate frame over an already-open socket (status flipped directly via the DB for this one, isolating the reauth-branch check from the eviction path the first test already covers); a frame larger than `WS_MAX_PAYLOAD_BYTES` closes the connection with code `1009`. `backend/tests/auth.test.js` gained a test proving a still-unexpired access token is rejected by `GET /api/auth/me` on the very next request after disable — previously only login and refresh were covered, neither of which exercises the token-already-in-hand case this entry closes.

### Cross-channel "Catch Me Up" workspace digest (2026-07-17)

Implemented from `FEATURE_REQUEST.md` backlog entry 6. Two deliberate scope reductions from the original design text, both flagged in the design itself as acceptable v1 fallbacks rather than guessed:

- **Explicit `channelIds` instead of persisted starring**: the design offered a `starred_channels` table as one option, "or starts with an explicit channel list" if that table doesn't exist yet. This pass takes the explicit-list fallback — the request body's `channelIds` is a caller-supplied, per-request list (validated against current channel membership, silently dropping anything the caller can't read), not a durable per-user preference. A first-class starring feature (schema, toggle endpoints, sidebar UI) is a separable concern, left for a future entry if requested. `includeMentionsOnly` was dropped from the request shape entirely — it would have been redundant with `channelIds` being empty, which already means "mentions only."
- **Single-completion truncation instead of hierarchical batching**: the design called for per-batch summaries merged into a final digest. This pass reuses the exact truncate-to-`maxInputChars` + one-completion pattern channel-summarize/thread-extract already use (`llm/aiService.js`'s `runStreamingCompletion`), rather than a new multi-call batching architecture — the source set (unread mentions plus a bounded set of explicitly chosen channels within a time window) is typically much smaller than a full channel history, so the existing truncation safety valve is a reasonable fit. The audit payload still reports a `chunkCount` field (always `1` in this pass) so a future batching pass can extend it without a payload-shape change.

- **Backend**: `backend/src/services/workspaceDigestService.js` selects unread, non-dismissed mentions (`mention_notifications`, filtered through a live `channel_members` join so a mention from a channel the caller has since lost access to is excluded — the same stale-private-channel-access concern the mentions-panel entry already fixed) plus messages from any caller-membership-validated `channelIds`, both scoped to the requested workspace and a bounded time window, deduplicated, capped, and sorted chronologically. New `POST /api/ai/workspace-digest` (`routes/ai.js`), behind `requireAuth` + `requireWorkspaceMember` (existence-hiding 404) + a new `aiDigestRateLimiter` (`llm/aiRateLimit.js`, stricter than `aiProxyRateLimiter` since a digest scans more than one channel per request). `sinceHours`/`sinceDays` are mutually-exclusive alternate units, bounded by a new env-only `AI_DIGEST_MAX_WINDOW_HOURS` (default 336h/14 days, not admin-editable — an operational safety cap, same class as `EMBEDDING_*` config). New `llm.digest_prompt_version` app_settings key (`digestPromptVersion`), mirroring `summaryPromptVersion`/`taskPromptVersion` exactly, with its own versioned template in `llm/promptTemplates.js` producing a markdown digest with fixed section headings (Urgent Mentions / Action Items / Unresolved Questions / Decisions Made) and per-line channel/author tags so the model can cite where each item came from. Audited as `AI_WORKSPACE_DIGEST_REQUESTED` with workspace id, window, selected/mention/channel-message counts, provider/model, prompt version, and truncated input length — never raw message content.
- **Cancellation**: `runStreamingCompletion` and both real adapters (`ollamaAdapter.js`, `vllmAdapter.js`) gained an optional `signal` parameter, additive and unused by the pre-existing summarize/extract-tasks call sites. ### Phase 2: Local Auth And API Base — complete (2026-07-12)

- **Password policy & hashing** (`backend/src/auth/passwordPolicy.js`): minimum 10 characters plus a ~30-entry common-password deny-list. `bcryptjs`'s async API only, salt rounds floored at 12 via config (`Math.max(12, ...)` — can't be configured below the floor).
- **JWT access tokens with rotation-ready key id** (`backend/src/auth/jwt.js`): 15-minute tokens signed with `JWT_SECRET`, carrying a `kid` header from `JWT_KEY_ID`. Verification checks both the signature and that `kid` matches the currently configured key id — catches not just a full secret rotation (which already invalidates the signature on its own) but the narrower case of `JWT_KEY_ID` being bumped without actually rotating the secret.
- **Refresh tokens with rotation + reuse detection** (`backend/src/auth/refreshTokens.js`, `database` unchanged — reuses Phase 1's `refresh_tokens` table): opaque random tokens, only a SHA-256 hash stored. Every `/api/auth/refresh` call revokes the presented token and issues a new one inside one row-locked (`forUpdate`) transaction. Presenting an *already-revoked* token — a replay — revokes every other outstanding token for that user and is audited as `AUTH_REFRESH_REUSE_DETECTED`.
  - **Real bug hit and fixed here**: the reuse-detection branch originally revoked the user's other tokens and then `throw`-ed the reuse error *inside the same `db.transaction()` callback* — which rolled back the revocation it had just performed, since a rejected transaction callback rolls back everything in it. A test (`replay the old token, then try to use the token issued by the legitimate rotation`) caught this: the "revoked" token still worked. Fixed by having the transaction callback always resolve with a discriminated result (`{kind: 'ok'|'reuse'|'invalid'|'expired', ...}`) and throwing the actual error *after* the transaction commits.
- **Cookie handling**: refresh token in an `httpOnly`, `SameSite=Strict` cookie (`Secure` in production), scoped to `/api/auth` specifically — tighter than the plan's minimum, since that's the only path range that ever needs it. Access token returned in the response body only, held in memory on the client (Phase 3 will wire this into the frontend).
- **Rate limiting** (`backend/src/auth/rateLimit.js`): stacked per-IP (20/15min) and per-username (10/15min) limiters on login, per-IP (10/hour) on signup. **Real bug hit and fixed here**: the test suite itself tripped these limits — many signups/logins within one test file share one in-process limiter, which isn't the credential-stuffing pattern the limiter exists to catch. Fixed with a `skip` condition on `NODE_ENV === 'test'` (set by `npm test`, never in dev/prod), not by weakening the limits themselves.
- **Anonymous audit actor** (`backend/src/audit/auditService.js`): added `ANONYMOUS_ACTOR_ID` (all-zero UUID, same convention as `GENESIS_HASH`) for audit events with no real authenticated actor yet — e.g. a failed login against a username that doesn't exist, where attributing the attempt to a real user's row would be wrong. `audit_logs.actor_id` has no FK constraint, so this is valid without a schema change.
- **Shared authorization module** (`backend/src/authz/membershipService.js`): `requireWorkspaceMember`/`requireWorkspaceAdmin`/`requireChannelMember`/`isChannelMember`, all taking `db` explicitly so they work identically inside a transaction or against the plain pool — built once here for both REST (now) and WebSocket handlers (Phase 3) to share, per Section 3. Adopted a specific status-code convention throughout: not authenticated → 401; not a member of a workspace/channel → **404**, not 403 (so a private resource's existence is never confirmed to someone who can't access it, including when the resource simply doesn't exist at all — both cases are deliberately indistinguishable); a member but lacking a specific privilege (e.g. non-admin action) → 403.

### Phase 3: Real-Time WebSockets And Layout UI — complete (2026-07-12)

- **WebSocket server** (`backend/src/ws/server.js`): attached to the same HTTP server as Express (`backend/src/index.js`, `http.createServer(app)` + `attachWebSocketServer`), on configurable `WS_PATH` (default `/ws`). Connections start unauthenticated — no joins, no data — until a client `authenticate` frame validates via the same `verifyAccessToken` Phase 2 already uses. Supports re-authenticating an already-open connection with a renewed token (rejects identity-switch attempts), and a periodic sweep (`WS_TOKEN_SWEEP_INTERVAL_MS`) disconnects any socket whose token has expired without a renewal frame arriving — the long-lived-connection requirement from Section 3.
- **Shared authorization, reused not reimplemented**: room joins call the exact same `authz/membershipService.js` from Phase 2 (`requireChannelMember`), re-validated on every `join` frame — including reconnects, since a fresh connection carries no memory of a prior session's joins. A non-member and a nonexistent channel get the identical error message, preserving the REST layer's "don't confirm private-resource existence" property over the socket too.
- **Message creation unified across transports** (`backend/src/services/messageService.js`): extracted from the Phase 2 REST route so REST and WebSocket sends run through identical validation/insert logic — the same anti-drift principle Section 3 requires for authorization, applied here to message creation. Both paths now also add `username` to the message payload (passed through from the already-decoded JWT/session, not an extra query) and the REST history endpoint joins `users` for the same field — added because displaying raw UUIDs as message authors would have been unusable; the history query needed every `created_at` reference qualified as `messages.created_at` once that join was added, since `users` also has a `created_at` column and the unqualified reference became ambiguous (caught by a test, not by inspection).
- **Presence engine** (`backend/src/ws/presence.js`, `ws/connectionRegistry.js`): in-memory only, per the Scalability Target's single-instance assumption. `online`/`away` are server-observed from heartbeat recency (never a client timestamp); a periodic sweep (`WS_PRESENCE_SWEEP_INTERVAL_MS`) downgrades stale connections to `away`, and a user's last connection closing broadcasts `offline` to every other authenticated client.
- **Rate limiting & connection caps** (`backend/src/ws/rateLimiter.js`, `config.ws`): a fixed-window per-user cap on message sends (`WS_MAX_MESSAGES_PER_WINDOW`, default 10/10s) and a per-user concurrent-connection cap (`WS_MAX_CONNECTIONS_PER_USER`, default 5) enforced at `authenticate` time.
- **Optimistic send + reconciliation**: the WS `message` frame accepts an opaque `clientNonce`, echoed back verbatim in the `message_created` broadcast — meaningless to other clients, but lets the sender's own UI match a confirmed message back to the placeholder it rendered before the round trip completed.
- **Backend test suite** (`backend/tests/{ws,presence}.test.js`, 24 new tests; 56 total across the whole backend now): a real `ws` client against a real listening server on an ephemeral port (no mocking) covering the unauthenticated-frame rejection, invalid-token rejection, re-authenticate-same-identity, reject-identity-switch, the concurrent-connection cap, non-member join denial with existence-indistinguishable errors, reconnect re-validating membership fresh (denied → added via REST → a **new** connection now succeeds, proving nothing was cached), message delivery to joined clients only (not to a non-joined outsider), REST-sent messages broadcasting to WS-joined clients, `clientNonce` echo, send-before-join rejection, rate-limit enforcement, and presence online/offline transitions; a separate fake-timers unit test (`presence.test.js`) deterministically proves the online→away staleness sweep without waiting out a real time window.
- **Two real bugs found by the test suite itself, not by inspection**:
  1. Several new WS/presence tests initially timed out waiting for a broadcast that had, in fact, already arrived — `await`ing the triggering action (a REST call, or another socket's own `authenticated` ack) *before* attaching the listener for the resulting broadcast left a window where the event could arrive and be silently dropped with nothing listening yet. Fixed by attaching every such listener before triggering the action that causes it, not after.
  2. Two WS tests failed for an unrelated, real reason: adding a user to a channel requires they already be a *workspace* member (Phase 2 rule), and the tests had skipped that setup step — fixed the tests, not the app, since the app was correctly enforcing its own rule.
- **Frontend** (`frontend/src/{api,context,ws,components}/`): `api/client.js` (in-memory-only access token, silent refresh-and-retry-once on a 401, never `localStorage`), `api/auth.js` + `AuthContext.jsx` (session restore on load via `GET /api/auth/me`, added this phase since the frontend has no other way to learn who's logged in after a bare token refresh), `ws/socket.js` (authenticate-frame handshake, exponential-backoff-with-jitter reconnect, periodic silent re-auth over the same open socket well before the 15-minute token expiry, heartbeats), and the 3-column UI itself: `LoginScreen`, `WorkspaceSidebar` (workspaces + channels, create/join), `ChannelView` (message feed with optimistic send, composer), `ThreadSidebar` (thread replies), `PresenceBadge`. All built from `global.css`'s copied design tokens (Section 7) — no new colors, fonts, or spacing introduced.
- **A third real bug, found only by driving the app in an actual browser** (Jest/supertest couldn't have caught this — it's React-lifecycle-specific): React 18 StrictMode double-invokes mount effects in development. `AuthContext`'s session-restore effect had no guard against this, so it fired `POST /api/auth/refresh` twice on mount. Since refresh tokens rotate-on-use (Phase 2), the second call raced the first: if it lost the race, it hit reuse detection and 401'd — and because that failure's `.catch(() => setStatus('anonymous'))` could resolve *after* the winning call had already set `status: 'authenticated'`, it could silently clobber a valid session back to the login screen. First-load-with-no-session-yet showed the underlying symptom (2 refresh attempts instead of 1) before this was diagnosed as a race rather than expected behavior. Fixed with a `useRef` call-once guard around the restore call, independent of how many times the effect body itself runs.
- **End-to-end validation actually run**: full backend Jest suite (56/56) against the live Postgres container; `docker compose up -d --build backend frontend` rebuilt both images; a real headless-Chromium session (Playwright, driven via a throwaway script — no `chromium-cli` available in this environment, so `playwright` was installed directly, reusing/updating the cached browser binary) drove the actual app end to end: sign up → create a workspace → create a channel → confirm the WebSocket join enables the composer → send a message and see it render → open a thread and send a reply and see it render → reload the page and confirm the session restores (username still shown, no re-login needed) — with `console --errors` checked at each step, not just the final screenshot. Screenshots sent to the user directly.
- **Not yet done**: the LLM proxy and admin audit dashboard remain as documented in Section 8 (Phases 4–5). Frontend automated tests (Vitest/RTL) are not set up — Phase 3 verification relied on the backend Jest suite plus real end-to-end browser driving, which is what actually caught the StrictMode race above; this is a real gap worth closing before the codebase grows much further, not a permanent decision.

### Production Deployment: whisper.silentlattice.dev live (2026-07-12)

Triggered by the user hitting `https://whisper.silentlattice.dev` and getting Silent Lattice's login page plus a TLS warning — the Phase 5 nginx/TLS work (Section 8) hadn't started yet; everything up to this point only ran in the local Docker Compose test environment. Diagnosed and fixed end-to-end, with explicit user confirmation before touching the shared production nginx (`wireservice-nginx-1`, which also serves live `silentlattice.dev`/`dev.silentlattice.dev` traffic).

- **Root cause of the original symptom**: no `server_name whisper.silentlattice.dev` block existed in `/root/wireservice/nginx/nginx.conf`, so nginx fell back to the first `listen 443 ssl` block (`silentlattice.dev`) for the unmatched hostname — explaining both the wrong page (that block's `proxy_pass`) and the TLS warning (that block's cert, mismatched against the requested SNI).
- **DNS was already fine** — `whisper.silentlattice.dev` already resolved (a wildcard record must already exist at the registrar); no GoDaddy change was needed this round.
- **`wireservice-nginx-1` is not Compose-managed**, despite `sl-admin.py`'s infra menu assuming it is (`docker compose ... nginx` — confirmed no `nginx` service exists in `/root/wireservice/docker-compose.yml`, so that menu's rebuild/restart/reload options are currently non-functional no-ops). It's a plain image (`wireservice-nginx`, built from `nginx/Dockerfile` which `COPY`s `nginx.conf` in at build time) run via a bare `docker run` with `--network wireservice_default --add-host host.docker.internal:host-gateway --restart unless-stopped -p 80:80 -p 443:443 -v /etc/letsencrypt:/etc/letsencrypt:ro`. Any future nginx.conf change needs a rebuild + recreate (or a `docker cp` + `nginx -s reload` for a faster, zero-port-downtime path — used here).
- **Cert**: issued via `certbot certonly --standalone -d whisper.silentlattice.dev` (user's choice — matches the existing two domains' method rather than migrating to `webroot`), requiring a brief `docker stop wireservice-nginx-1` to free port 80. Expires 2026-10-10.
- **A real architecture correction made live**: the original Section 2 design assumed Silent Whisper's containers would be reached via `host.docker.internal` on their published ports, mirroring `wireservice-dev`. That doesn't work — those ports are bound to `127.0.0.1` (deliberately, for security), and `host.docker.internal` traffic arrives via the Docker bridge gateway, a different source address than localhost, which a loopback-only bind rejects. `dev.silentlattice.dev`'s equivalent ports are bound to `0.0.0.0`, which is why that pattern works for it and didn't for Silent Whisper. Fixed by attaching `backend` and `frontend` to `wireservice_default` (declared as an `external` network in `docker-compose.yml`, not a manual one-off `docker network connect` — confirmed this survives a full `docker compose up --build` recreate) and having nginx address them by container name instead. **Section 2's design note has been superseded by this** — it now documents the actual working pattern, not the originally-assumed one.
- **Two more real bugs found while verifying, both fixed**:
  1. Nginx caches upstream DNS resolution for a `proxy_pass` hostname at reload/start time — recreating `backend`/`frontend` containers gives them new IPs, and nginx kept sending traffic to the old (now-dead) ones until reloaded again. `nginx -s reload` after any Silent Whisper container recreation is now a documented required step (RUNBOOK.md).
  2. Vite's dev server rejects requests with an unrecognized `Host` header by default (DNS-rebinding protection) — nginx forwards the real `Host: whisper.silentlattice.dev`, which Vite refused with a 403 until added to `server.allowedHosts` in `vite.config.js`.
  3. Vite's own HMR (hot module reload) WebSocket client tries to connect over the public domain too and fails (`Unexpected response code: 200` — nginx's `/` location has no `Upgrade`/`Connection` headers, unlike `/ws`). This is cosmetic console noise from the dev server's own convenience feature, not a break in the application (the app's own `/ws` endpoint was directly verified working). Left unfixed for now — the real fix is serving a production static build instead of a dev server behind the public domain, which is Phase 5 territory, not a quick config tweak.
- **A significant finding, not caused by this work but affecting it**: `/etc/letsencrypt/renewal-hooks/pre/stop-nginx.sh` and `post/start-nginx.sh` both run `docker compose stop/start nginx` from `/root/wireservice` — but, per the point above, there is no `nginx` service in that Compose file. These hooks are silently non-functional, meaning **none of the three certs (including the two pre-existing ones) will actually free port 80 for `certbot renew`'s standalone authenticator**, and all three are liable to fail to auto-renew around 60 days from issuance. This predates this session's changes and affects `silentlattice.dev`/`dev.silentlattice.dev` too, not just the new cert — flagged to the user, not fixed yet (fixing it means rewriting those hook scripts to `docker stop`/`docker start wireservice-nginx-1` directly, since the container isn't Compose-managed).
- **End-to-end validation actually run**: `curl` against `/health` and `/` with the real cert inspected via `openssl s_client` (confirms `CN = whisper.silentlattice.dev`, valid dates); a raw Node `ws` client completing a full `authenticate` round trip through nginx to confirm the WebSocket upgrade path works, not just assumed from the location block; confirmed `silentlattice.dev` and `dev.silentlattice.dev` still return 200 (no regression to live traffic); a full Playwright browser session against `https://whisper.silentlattice.dev` itself (not localhost) repeating the entire Phase 3 flow — signup, workspace, channel, live message, thread reply, page reload with session restore — end to end over the real public URL with a trusted cert.
- **Not yet done**: certbot renewal hooks (flagged above, needs a decision + fix); Vite HMR-over-public-domain noise (cosmetic); no production static frontend build yet (still a dev server, acceptable for now but not ideal long-term for a public URL).

### Post-Phase-5 finding: no API to add a user to a workspace (2026-07-12)

Discovered provisioning real login credentials on request — the only user in the database at that point was `auditadmin5`, leftover debris from a Jest test run that never got cleaned up (deleted). ### Self-service password change (2026-07-12)

Implemented from `FEATURE_REQUEST.md`'s ranked entry 3, on request. The design was written implementation-ready in advance (`POST /api/auth/change-password`, revoke-all-then-reissue token handling, `AUTH_PASSWORD_CHANGE` audit type), so this entry records what was actually built and verified, not a fresh design pass.

- **`POST /api/auth/change-password`** (`backend/src/routes/auth.js`), behind `requireAuth` and a new `changePasswordLimiter`: `currentPassword` checked with `bcrypt.compare` against the caller's own `password_hash` (401 `UnauthorizedError` on mismatch — a credential check, not a validation failure); `newPassword` validated with the existing `assertValidPassword` (same policy as signup, no separate rules); new hash written via `bcrypt.hash` at `config.auth.bcryptSaltRounds`.
- **Session handling**: calls the existing `revokeAllRefreshTokensForUser(db, userId)` (`auth/refreshTokens.js`, already used by refresh-reuse detection) to revoke every outstanding refresh token for the account — including the request's own current one — then immediately issues a fresh access token + a fresh refresh token in the same response, exactly the shape `/login` returns. The net effect: every *other* session is forced to re-authenticate, while the tab that just changed the password keeps working uninterrupted, since it's handed a brand-new valid pair rather than having its old one silently revoked out from under it.
- **Rate limiting** (`backend/src/auth/rateLimit.js`'s new `changePasswordLimiter`): per-user (not per-IP, copying `llm/aiRateLimit.js`'s `aiProxyRateLimiter` shape — `keyGenerator: (req) => \`change-password:${req.user.id}\``), 10/15min, same ceiling as `loginUsernameLimiter`'s credential-guessing budget. Not explicitly called out in the original `FEATURE_REQUEST.md` design, but required by `CLAUDE.md`'s "rate-limit authentication ... endpoints — not optional, not a later phase": a short-lived stolen access token would otherwise be an unlimited number of `currentPassword` guesses against the real account owner.
- **Frontend**: `ChangePasswordPanel.jsx` (new, `frontend/src/components/`), copying the `AiSettingsPanel` modal pattern exactly (`backdrop`/`panel`/`header`/`title`/44px `closeButton`/`subtitle`, inline `error`/`saving` state). Reachable from a "Change Password" control next to "Sign out" in `WorkspaceSidebar`'s `userRow` — visible to every user, not admin-gated, unlike the AI Settings/Audit Log/notification-permission controls already in that row. `userRow` already needed `flexWrap` (added for the notification-permission button in the mentions feature) to hold a fourth control without repeating the sidebar's earlier known overflow bug. `api/auth.js`'s new `changePassword()` follows `login`/`signup`'s exact pattern (`apiFetch` + `setAccessToken` on the response), and `AuthContext.jsx` exposes it alongside `login`/`signup`/`logout` — no `user`/`status` state change needed on success since it's the same account, just a fresh token.
- **A real test-authoring bug caught by the test itself, not a feature bug**: the first version of the "current session survives, other sessions don't" backend test asserted the *old* (pre-change) refresh token was rejected *before* asserting the newly issued one still worked. Replaying an already-revoked refresh token correctly trips reuse detection (`auth/refreshTokens.js`), which itself revokes every outstanding token for the user — including the one the test had just gotten back from the change-password response — so the test's own assertion order was clobbering the very thing it was trying to prove. Fixed by checking the new token first; the underlying endpoint behavior was correct throughout, confirmed independently with a standalone debug script against the real database before concluding this.
- **Test suite**: 5 new backend tests (`backend/tests/auth.test.js`) — wrong `currentPassword` → 401 with the old password still working; policy-violating `newPassword` → 400; success → new tokens work, old password rejected, `AUTH_PASSWORD_CHANGE` audited; the current session's freshly issued refresh token works while the pre-change one is revoked; unauthenticated request → 401 — 136/136 backend tests passing. 1 new e2e test (`frontend/e2e/workflows.spec.js`'s `change password` describe: wrong-password inline error, then a full change → sign out → old password rejected → new password succeeds round trip through the real UI) — 12/12 e2e tests passing, verified in the same two signup-rate-limit-budget-safe batches (9 + 3) this file's mentions entry already established as the working pattern, restarting the backend between them to clear `signupIpLimiter`'s in-process state.

Playwright picked the sidebar button (first in DOM order), which sits behind the open modal's backdrop and so could never actually be clicked. Fixed by scoping the test's locator to `button[type="submit"]:has-text(...)`, which the two buttons' `type` attributes already made unambiguous without changing any component code.

### Admin dashboard: user provisioning, role assignment & password reset (2026-07-12)

Implemented from `FEATURE_REQUEST.md`'s ranked entry 3, on request. The design reused `requireAnyWorkspaceAdmin`'s established "no global admin role, ADMIN-in-at-least-one-workspace" ceiling and read the request as three per-workspace admin actions, not a platform-wide user-management system — same interpretation the design entry already flagged explicitly.

### Incident: the test suite was deleting real user data (2026-07-12)

While rebuilding the `admin`/`user` accounts to verify the new invite endpoint, a routine `npm test` run to confirm the endpoint didn't break anything **silently deleted both accounts** from the live database. Root cause: `backend/tests/helpers/resetDb.js`'s `beforeEach` hook (`await db('users').del()`, unconditional) had, since Phase 1, always run against whatever database `backend/.env`'s `PGDATABASE` pointed at — the same `silent_whisper` database the actual deployed backend connects to, because host-run tests and the Docker Compose stack share one `.env`-configured Postgres instance on this host. This had never surfaced as a problem before simply because there was never anything real in that database to lose; provisioning real credentials for the user was the first time this test-suite behavior collided with something worth protecting.

**Fixed properly, not just documented**: added an isolated `silent_whisper_test` database on the same Postgres instance. `backend/package.json`'s `test` script now sets `PGDATABASE=silent_whisper_test` as a shell-level environment variable, *before* Node starts — every file that reads `process.env.PGDATABASE` (`src/config.js`, and the few test files with their own direct `pg`/Knex connections: `resetDb.js`, `auditService.test.js`, `auditDashboard.test.js`) picks it up automatically and identically, since dotenv's default behavior never overwrites an env var that's already set. No per-file code changes were needed — the fix is exactly one line in `package.json` plus creating and migrating the new database (`npm run migrate:test-db`, new script, same `knexfile.js`). The grants migration (`0007_grants.js`) already handled a pre-existing `app_runtime_user` role correctly (checks `pg_catalog.pg_roles` and `ALTER ROLE`s instead of erroring on `CREATE ROLE`), since Postgres roles are cluster-wide, not per-database — so it needed no changes to work against the new database.

Verified the fix, not just implemented it: ran the full suite twice against `silent_whisper_test` (120/120, then 119/120 with the one already-known unrelated flake) and confirmed after each run that `admin`/`user` remained present and untouched in the real `silent_whisper` database. The two real accounts were recreated exactly as before (same usernames, same passwords) using the real signup API, and the workspace membership this time went through the real new invite endpoint end to end instead of the direct-DB workaround from the entry above.

**Not yet done**: this fixes local host-run testing, which is the only way tests are currently run. If CI or any other automation is added later that might point at a different `PGDATABASE` value, re-verify this same isolation holds there too — the protection here is "the test script sets the database name," not any deeper guarantee inside the test code itself, so a differently-invoked test run (e.g., `NODE_ENV=test node ...` directly, skipping the `test` npm script) would not be protected.

### Semantic message & channel search via pgvector (2026-07-12)

Touches the DB image, schema, the provider-adapter boundary, a new async ingestion pipeline, a new API surface, and the frontend; the largest single feature shipped so far. - **Provider-adapter boundary extended, not duplicated** (`llm/adapters/{ollama,vllm,disabled}Adapter.js`): all three gained `embed({settings, text})`, the same one-branch-point rule `generate`/`checkHealth` already establish. Ollama's native `/api/embeddings`, vLLM's OpenAI-compatible `/v1/embeddings`; both validate the returned vector's length against `settings.dimension` and throw a clear `UpstreamError` on mismatch rather than letting a misconfigured `EMBEDDING_MODEL` surface as an opaque Postgres vector-dimension error at insert time. - **Async, failure-tolerant ingestion** (new `backend/src/search/`): `embeddingConcurrencyGate.js` (`llm/concurrencyGate.js`'s identical shape, a separate counter so embedding backlog and channel-summary generation can't starve each other); `embeddingService.js`'s `embedText(db, text)` (shared by both the worker and the search route's query-embedding step); `embeddingQueue.js`'s `enqueueEmbeddingJob` (a fire-and-forget-on-failure insert, called as a sibling step at both message-creation call sites — `routes/messages.js`'s POST handler and `ws/server.js`'s `handleMessage`, immediately after the existing `extractMentionedUserIds` call, the exact same "side effect, not part of message creation" pattern mentions already established); `embeddingWorker.js` (a `setInterval` sweep, same shape as `llm/healthCheck.js`/`ws/presence.js`, claiming up to `EMBEDDING_WORKER_BATCH_SIZE` pending rows per tick via `UPDATE ... WHERE id IN (SELECT ... FOR UPDATE SKIP LOCKED) RETURNING`, retrying on failure up to `EMBEDDING_MAX_ATTEMPTS` before dead-lettering).
- **Search API** (`routes/search.js`): `POST /api/search/semantic`, behind `requireAuth` + a new `semanticSearchRateLimiter` (30/5min per user, `llm/aiRateLimit.js` — higher than `aiProxyRateLimiter`'s 10/5min since a search is cheaper than a full generation). DM/group-DM hits are correctly authorized and navigable but don't highlight in the sidebar's workspace list — a pre-existing gap (no DM-browsing UI exists anywhere in this app yet) this inherits rather than introduces.
- **Four real bugs found by testing, not by inspection** — one of them a genuine SQL correctness bug, not just a test artifact:
  1. **`claimBatch`'s original query was wrong, not flaky.** It used the commonly-cited `UPDATE t SET ... WHERE id IN (SELECT ... LIMIT n FOR UPDATE SKIP LOCKED)` form. The `embeddingWorker.test.js` "claims at most `EMBEDDING_WORKER_BATCH_SIZE` pending jobs per tick" test failed intermittently — first dismissed as environment/scheduling flakiness (it passed 6/6 on immediate re-runs), which was itself a mistake worth recording: a re-run passing isn't evidence a race was ruled out, and this got caught later the same session when the *same* test failed again, in isolation, with nothing else running. Reproduced deterministically outside the app entirely, directly in `psql`: inserted 5 rows sharing one `created_at` (the realistic trigger — a batch `INSERT ... SELECT` evaluates `now()` once for the whole statement, but any exact tie on the `ORDER BY` column has the same effect), and the `IN`-subquery form's `UPDATE` claimed **all 5** instead of 3, confirmed via `EXPLAIN` to be planned as a `Nested Loop Semi Join` — an uncorrelated `IN`-subquery is free to be evaluated more than once per statement, and with tied sort values each re-evaluation can pick a different arbitrary N-of-many subset. Fixed with the standard, verified-correct alternative: a `WITH claimed AS (SELECT ... LIMIT n FOR UPDATE SKIP LOCKED) UPDATE ... FROM claimed WHERE ...` — a data-modifying/`FOR UPDATE`-containing CTE is never inlined by Postgres's planner, so it's guaranteed to execute exactly once. Confirmed the fix directly the same way: 5/5 clean `psql` runs claiming exactly 3, then 5/5 clean Jest runs of the same test, then two full backend-suite runs (196/197 both times, only the pre-existing unrelated flake below).
  2. `index.js`'s `start()` originally called `startEmbeddingWorker(db)` unconditionally, same as the other sweeps. Several existing test files (`ws.test.js`, `mentions.test.js`, `workspaceArchiving.test.js`) call `start()` directly for a real listening WebSocket server *without* mocking `global.fetch` (unlike the AI-route tests, which mock fetch but never call `start()` at all) — at this worker's 2s default poll interval, an un-gated auto-start would make real, repeated network calls against the live Ollama instance for the whole duration of every one of those unrelated test files. Fixed with a `config.nodeEnv !== 'test'` guard at the sweep-start call site, the same skip-under-test instinct `llm/aiRateLimit.js`/`ws/rateLimiter.js` already apply via `skip`/`skipInTest` — worker logic itself stays fully covered by `embeddingWorker.test.js`, which calls `runEmbeddingWorkerTick` directly with `fetch` mocked.
  3. The WS ingestion test (`embeddingIngestion.test.js`) initially asserted a job row existed immediately after receiving the `message_created` WS ack — failed intermittently, because the handler broadcasts that ack *before* it finishes awaiting the sibling `enqueueEmbeddingJob` call (correct, real behavior: broadcast isn't gated on that side effect), so a client can see the ack before the job row exists. Fixed by polling briefly instead of asserting synchronously on the first check, the same "don't assume a server-side side effect is done just because the client saw an ack" lesson `ws.test.js`'s own bug-fix history (Phase 3 entry above) already documents.
  4. **A separate, real e2e-test-only bug**: the semantic-search e2e test's result locator was an unscoped `text=...` match — since the seeded messages' content is already visible in the live channel feed rendered *behind* the search modal, the locator ambiguously resolved to that hidden copy instead of the actual result row, and clicking it hung for the full test timeout with Playwright reporting "element intercepts pointer events" (the modal backdrop, correctly sitting on top). Fixed by adding `aria-label="Search results"` to the results container (a real accessibility improvement, not just a test hook) and scoping the e2e locators to it.

- **Two real bugs found by testing, not by inspection, both in the new components rather than anywhere already-shipped**:
  1. `Menu.jsx` originally closed itself on any `window` `scroll` event (capture phase), reasoning that a popover positioned via a captured `getBoundingClientRect()` snapshot goes stale once anything scrolls. ### Basic Markdown formatting in messages (2026-07-13)

Implemented from `FEATURE_REQUEST.md`'s ranked entry, on request, following that entry's design closely. Client-side only, as designed — `messages.content` and the wire format are completely unchanged; `backend/src/validation.js`'s `assertMessageContent` needed no edit, and no backend test needed updating.

- **New tokenizer** (`frontend/src/markdown.jsx` — `.jsx`, not the design note's suggested `.js`: Vite's default esbuild loader rejects JSX syntax in a plain `.js` file outright, caught immediately by a real build failure the first time the file was actually imported, not by inspection). A hand-rolled multi-pass tokenizer, not a markdown library + sanitizer, exactly as designed — zero new runtime dependencies. `renderMessageContent(content)` replaces `ChannelView.jsx`'s old module-private `renderContentWithMentions` as the single content-rendering entry point, now also used by `ThreadSidebar.jsx` for both the thread root and every reply — closing a real, pre-existing gap found while wiring this up: thread replies never highlighted `@mentions` at all before this, since `ThreadSidebar.jsx` rendered `r.content`/`rootMessage.content` raw with no processing whatsoever.
- **Pass order matches the design exactly**: links (markdown-syntax `[label](url)`, then bare-URL autolink), bold (`**`, then `__`), italic (`*`, then `_`), mentions — each pass built on a shared `applyPass(nodes, regex, toNode, nextKey)` primitive that only re-scans the plain-text segments an earlier pass left behind, never touching an already-tokenized element.
- **Link safety**: `isSafeHttpUrl` mirrors `validation.js`'s `assertHttpUrl` scheme check client-side (`new URL(value).protocol` must be `http:`/`https:`). A markdown-syntax link pointing at an unsafe scheme renders its label as plain text (consumes the token, never leaks raw `[…](…)` source); a bare autolink can't match an unsafe scheme by construction (the regex requires an explicit `http(s)://` prefix) but is still scheme-checked for defense in depth. Every real link gets `target="_blank" rel="noopener noreferrer"`.
- **Four real bugs found by writing tests, not by inspection, before this ever reached a browser**:
  1. **Mentions never actually highlighted inside bold/italic text.** The first implementation had `boldToNode`/`italicToNode` wrap their captured text as a bare string child; since `applyPass` only re-scans array entries that are still plain strings, and a `<strong>`/`<em>` element's own children are never walked back into by a later pass, the mention pass (which runs last) could never reach inside one. Fixed by having `boldToNode`/`italicToNode` run the mention pass on their own captured content before wrapping it, and re-threading `nextKey` as the counter *function* itself (not a pre-computed value) through every `toNode` call, since bold/italic now need to mint more than one key per match.
  2. **`__word__`-shaped Python dunders (`__init__`, `__del__`) got read as bold.** The outer `(?<!\w)`/`(?!\w)` boundary guard (added specifically to keep `_`/`__` from triggering inside snake_case identifiers, e.g. `my_file_name`) only rejects a delimiter touching a word character *outside* the pair — it has nothing to say about "call __init__ then", where both `__`s are correctly flanked by spaces on their outside and the token is genuinely structurally identical to real single-word emphasis. Resolving this fully needs CommonMark's actual delimiter-flanking algorithm, more than "basic" scope calls for; fixed with a narrower, easy-to-verify rule instead — reject an underscore-delimited match whose captured content is a single bare identifier with no internal whitespace. The real cost: `__oneword__`/`_oneword_` no longer bolds/italicizes a single word (asterisks remain available and unaffected for exactly that case, since they don't share this ambiguity) — a deliberate, narrow trade-off, not an oversight.
  3. **A URL containing its own parens got truncated, leaving a stray `)` in the rendered text.** `MD_LINK_RE`'s original URL group (`[^\s)]+`) stopped at the *first* `)`, so `[here](javascript:alert(1))` parsed the URL as `javascript:alert(1` (missing its own closing paren) and left the link syntax's real closing paren behind as a dangling literal character — caught by the e2e test's exact-text assertion on the unsafe-scheme case, not a passing "it renders as text" assertion that would have missed the stray character. This wasn't only a contrived-XSS-example problem — the identical truncation hits any legitimate URL with internal parens (a Wikipedia-style `.../Foo_(disambiguation)`). Fixed by widening the URL group to `[^\s]+` (greedy, still bounded by whitespace either way) relying on backtracking to land on the *last* `)` before whitespace rather than the first.
  4. **No frontend unit-test runner existed to write tokenizer tests against at all** — confirmed by inspection (`frontend/package.json` had no Vitest/Jest, only Playwright e2e), matching a gap the prior HIG-overhaul entry's log had already flagged in passing. Added Vitest (`frontend/vitest.config.js`, `@vitejs/plugin-react` wired in for the same JSX-runtime reason the app's own `vite.config.js` needs it — its absence surfaced immediately as `ReferenceError: React is not defined` the first time a test actually invoked a tokenizer function) plus a `test:unit` script, scoped to `src/**/*.test.{js,jsx}` specifically since Vitest's default include glob also matches `e2e/workflows.spec.js` and errors trying to import a Playwright spec as its own. This is now available for the `Menu.jsx` component-level coverage the previous entry flagged as a natural companion, not just this one.
- **Test suite**: 15 new Vitest unit tests (`frontend/src/markdown.test.jsx`) covering bold/italic via both delimiter forms, the dunder/snake_case rejection cases, markdown-syntax and bare-URL links, the internal-parens fix, unclosed/malformed syntax falling back to literal text, and a mention surviving one level of nesting inside bold. 2 new e2e tests (`workflows.spec.js`'s `markdown formatting` describe): a real message typed through the composer and sent renders genuine `<strong>`/`<em>`/`<a href rel="noopener noreferrer" target="_blank">` elements in the feed, plus the `javascript:` case rendering as plain text with no stray characters; a second test confirms the mention-inside-bold fix holds through the real WebSocket send → render path, not just the tokenizer in isolation. All existing e2e tests re-run clean (24 total across three signup-budget-safe batches — this entry's 2 new tests pushed the full-suite signup count from 22 to 24; `RUNBOOK.md` updated). Backend suite unaffected, as expected for a frontend-only entry (196/197, the same pre-existing unrelated flake).

### Enterprise authorization model, slice 2: invitations + organization routes (2026-07-14)

Implemented the second of four planned slices of `FEATURE_REQUEST.md`'s ranked entry 1, per its own sequencing (schema/permissions/authz cutover → **invitation/org routes** → frontend → full test-suite migration). This slice is backend/API-only — no invite-redemption page, org switcher, or create-org UI — and self-service signup deliberately stays open (invitations are an *additional* account-creation path, not a replacement); closing signup and migrating the 14-of-24 backend test files that call `signup()` directly remains deferred to a later slice. Entry 1 stays **Proposed**, not Done.

Three scoping decisions were confirmed with the user before implementation, all locked into the design below: signup stays open (with a consequence — every fresh signup now needs an `organization_members` row); no frontend work this slice; `POST /workspaces`/`GET /workspaces/discoverable` become org-aware with a backward-compatible default (an optional `organizationId`, resolved to the caller's sole org membership when omitted, an explicit 400 only once a caller genuinely belongs to more than one — which slice 2 itself makes possible for the first time, since creating an org auto-enrolls the creator into it alongside their existing default-org membership).

Design:
- **Migration `0014_invitations.js`** — one `invitations` table (`scope_type` ORGANIZATION/WORKSPACE, `organization_id`/`workspace_id` nullable with exactly one set per scope, `email`, `invited_role`, `invited_by`, `token_hash` following `refresh_tokens`' never-store-raw convention exactly, `status` PENDING/ACCEPTED/REVOKED as a real stored column with expiry checked live at `expires_at < now()` rather than a stored EXPIRED value or a sweeper job — mirroring how `refresh_tokens` itself handles expiry), a unique index on `token_hash` (the entire hot path for the two public lookup routes), and partial indexes on `organization_id`/`workspace_id`.
- **`permissions.js`** gained `ORG_INVITE`/`ORG_MANAGE_MEMBERS` and an `ORG_ROLE_PERMISSIONS` map (`ORG_ADMIN` gets both, `ORG_MEMBER` neither — no owner-uniqueness tier the way workspaces have `OWNER`). Workspace-invitation creation deliberately reuses `WORKSPACE_MANAGE_MEMBERS` rather than adding a new `WORKSPACE_INVITE` permission — nothing yet differentiates "can invite" from "can manage the roster," so a single-consumer permission constant would be unused abstraction; add it only once something actually needs the split.
- **`membershipService.js`** gained `getOrgRole`/`requireOrgMember`/`requireOrgPermission`, mirroring `requireWorkspacePermission`'s shape function-for-function, including the system-admin-override-checks-org-existence-not-membership behavior. Org creation itself (`POST /organizations`) is gated directly on `isSystemAdminUser`, deliberately **not** routed through `requireSystemPermission`'s OR-fallback — that fallback exists narrowly to avoid locking out existing workspace admins from AI-settings/audit during slice 1's transition, and extending it to "any workspace admin can create an organization" would have been a real, unintended privilege widening rather than a continuation of its actual purpose.
- **New `backend/src/routes/organizations.js`** — `POST /` (system-admin only, auto-enrolls the creator as `ORG_ADMIN` transactionally, since an org with zero members would be an unmanageable dead end), `GET /` (caller's own orgs via a join, or every org if system admin), full member roster/add/role-change/remove (`DELETE` is genuinely new here — no workspace-side equivalent exists yet, but the design explicitly calls for org member removal). Two deliberate deviations from the workspace-member pattern, both locked-in design decisions: **no cascade** on member removal (org and workspace membership stay independent — removing someone from an org never touches their `workspace_members` rows) and **no "≥1 admin" invariant** (unlike the workspace `OWNER` guarantee, an org can end up with zero `ORG_ADMIN` members; the system-admin override remains the escape hatch).
- **New `backend/src/auth/invitationTokens.js`** — token generation/hashing factored out as its own small module (identical algorithm to `refreshTokens.js`'s `generateRawToken`/`hashToken`) since both the org/workspace creation routes and the redemption router need it without reaching into each other's internals. `INVITATION_TOKEN_TTL_MS` (7 days) is a fixed constant, not env-configurable, unlike `refreshTokenTtlMs`.
- **New `backend/src/routes/invitations.js`**, mounted at `/api/invitations` — follows `auth.js`'s per-route public/authenticated pattern (no router-wide `requireAuth`), not `workspaces.js`'s router-wide one, since it mixes public routes (`GET /:token`, `POST /:token/accept`) with an authenticated one (`POST /:id/revoke`). `GET /:token` gives the same generic 404 for "doesn't exist" / "revoked" / "already accepted" / "expired," extending this codebase's existence-hiding instinct to a public, token-keyed surface instead of a membership check. `POST /:token/accept` is one row-locked (`forUpdate`, same precedent as `rotateRefreshToken`) transaction: re-validates the token, checks the same username/email collision signup uses, creates the account, attaches the invited-role membership, marks the invitation `ACCEPTED`, and logs the new user in via the exact same `issueRefreshToken`/`setRefreshCookie` path signup uses — `issueRefreshToken` already accepted an optional transaction parameter (`trx = db`), so no changes were needed there at all. `setRefreshCookie` had to be exported from `auth.js` (previously private/module-scoped) rather than duplicated, to keep the cookie shape defined in exactly one place. A workspace-scoped acceptance also auto-enrolls into the earliest-created org, the same signup-parity reasoning as the next bullet.
- **`auth.js`'s `POST /signup`** — wrapped in a transaction (it wasn't one before) and now also inserts an `organization_members` row into the earliest-created organization (`ORDER BY created_at ASC`, explicit and deterministic — the pre-slice-2 workspace-creation shim's unordered `.first('id')` would have become nondeterministic the moment a second org could exist, which this slice is what actually makes possible for the first time). No new audit event — folded silently into the existing `AUTH_SIGNUP` row, the same precedent `POST /workspaces` already sets by not emitting a separate event for the creator's own membership insert.
- **`workspaces.js`** — `POST /` and `GET /discoverable` both gained an optional `organizationId`, resolved by a new local `resolveCallerOrganization` helper (deliberately kept local to this file, not promoted into `membershipService.js` — it's a workspace-creation-specific convenience around "caller's sole org," not a general authorization primitive; `organizations.js`'s own routes always take an explicit `:orgId` and never need this resolution). Confirmed as a true no-op for every account that existed before this slice: with exactly one org membership per user, the resolution always lands on that same org, identical in outcome to the old shim. New `POST /:workspaceId/invitations`, mirroring the org-scoped creation route.
- **New audit action types**: `ORGANIZATION_CREATED`, `ORGANIZATION_MEMBERSHIP_CHANGE`, `INVITATION_CREATED`, `INVITATION_REVOKED`, `INVITATION_REDEEMED`.
- **`rateLimit.js`** gained `invitationCreateLimiter` (per-actor, same 20/hour shape as `adminUserCreateLimiter`). Redemption reuses `signupIpLimiter` directly rather than a fourth signup-shaped limiter — accepting an invitation *is* a signup from an abuse standpoint. The context-preview `GET /:token` and the authenticated, permission-gated `POST /:id/revoke` need no limiter of their own, matching this codebase's existing precedent for cheap indexed reads and already-gated state-flips.
- **`resetDb.js`** — `invitations` added to the cleanup list, positioned before `workspaces`/`users` (a real FK dependency with no cascade specified). `organizations` stays uncleared, as it already was in slice 1 — extended reasoning, not a new decision: nothing asserts an exact global row count, and the earliest-created row must never be deleted anyway, since both signup and invitation redemption now depend on it staying stable across a test run.
- **Tests**: new `organizations.test.js` (creation/403/auto-enrollment, roster/add/role-change/remove with existence-hiding, the no-cascade and no-admin-invariant behaviors explicitly verified against real data, not just asserted by comment), `invitations.test.js` (org- and workspace-scoped creation, preview, accept — including the auto-login response shape, email collision, token reuse, and expiry — and revoke, including the inviter-vs-permission-holder-vs-outsider distinction), `workspaceOrganizations.test.js` (the `organizationId` resolution logic on both `POST /workspaces` and `GET /discoverable`, including the ambiguous-multi-org 400 and the org-isolation "doesn't leak" case), and an extension to `auth.test.js` asserting every fresh signup lands in the earliest-created org. Two real test bugs found and fixed during this pass, not app bugs: an org-creation test asserted `GET /organizations`' per-org `role` field using a caller who was *also* a system admin, which unconditionally hits the sees-every-org branch (`role: null` by design) rather than the join branch being tested; and a discoverable-scoping test queried using the workspace's own creator, whose token is always excluded from that workspace's own discoverable results by design (already a member) — both fixed by using a distinct, non-privileged caller instead of the actor that happened to be on hand.
- **Verification**: 260/261 backend tests passing (the one failure is `aiRoutes.test.js`'s pre-existing task-extraction audit-timing flake, confirmed non-deterministic by running it three times in isolation back-to-back — two passes, not caused by anything touched this slice, which never modified `ai.js` or its streaming path). Migration and full stack rebuilt and restarted against the real `silent_whisper` database (a `pg_dump` backup taken first). A complete live-API walkthrough: bootstrapped a system admin via `grant-system-admin.mjs` (slice 1), created a second organization (auto-enrolling the creator as `ORG_ADMIN`), created a `DISCOVERABLE` workspace explicitly inside it, confirmed a caller in the original default org sees an empty discoverable list (zero leakage), created and previewed an org-scoped invitation, redeemed it as a brand-new account (received a working access token + refresh cookie, same shape as signup), revoked a second invitation and confirmed the revoked token can no longer be accepted (404), and confirmed an ordinary signup still lands the new account in the earliest-created org.
- **Not in scope, per the design's own sequencing**: closing self-service signup and the 14-file test-helper migration, the frontend `hasPermission()` helper / org switcher / create-org UI / invite-redemption page, `managers_can_archive` enforcement — all later slices.

### Enterprise authorization model, slice 3: frontend (2026-07-14)

Framed as a frontend slice, but three small, additive backend companion changes were required for the frontend to function correctly and were made in the same commit rather than deferred, per the same reasoning slice 2 used for its own `resolveCallerOrganization` helper: shipping the frontend that consumes them without the fields/routes themselves would leave an intermediate state nothing could actually use. Closing self-service signup and the 14-file test-helper migration remain out of scope, per the design's own sequencing; entry 1 stays **Proposed**, not Done.

Four scoping decisions were confirmed with the user before implementation:

- **`isSystemAdmin` is exposed** on `GET /me`/`POST /login`/`POST /signup`/`POST /change-password`/`POST /invitations/:token/accept` — the column already existed and was already read server-side (`membershipService.js`'s `isSystemAdminUser`), this just surfaces it, read-only, no new authz surface. A structural no-op for every account with exactly one org (the overwhelming majority today), the same continuity property slice 2 established server-side for `resolveCallerOrganization`.

### Cancel AI summarize/extract-tasks on client disconnect, and nonce-delimited/JSON-serialized LLM prompts (2026-07-19)

Two independent Medium-severity findings from `docs/reviews/security-performance-review-2026-07-19.md` (`FEATURE_REQUEST.md` entries 2 and 4, originally ranked entries 2 and 4) implemented together in one pass since neither touches the other's code.

`POST /channels/:channelId/ai/summarize` and `POST /messages/:messageId/ai/extract-tasks` (`backend/src/routes/ai.js`) now wire the same `res.on('close', () => controller.abort())` → `AbortController` → `signal` pattern already shipped on `POST /ai/workspace-digest`, passing `signal` through to `runStreamingCompletion()`. With `LLM_MAX_CONCURRENT_REQUESTS` at its test/CPU-only default of 1, a closed tab or superseded retry no longer holds the sole concurrency slot for up to `LLM_TIMEOUT_MS` regardless — `concurrencyGate.js`'s `release()` (already shipped for the FIFO queue) hands the freed slot directly to the next waiter the moment the disconnecting request's provider call is aborted, rather than that waiter staying queued behind a dead connection until it times out on its own. No change to the normal (non-disconnected) path, and no new audit event — the existing `AI_SUMMARIZE_REQUESTED`/`AI_TASK_EXTRACTION_REQUESTED` events are unaffected, since a disconnected request never reaches its own `appendAuditEvent` call (the same "respond first, audit after" ordering every other AI route already has).

`backend/src/llm/promptTemplates.js`'s summary/task/digest prompt builders gained a "v2" template alongside the original "v1" (kept byte-for-byte unchanged, still reachable as an explicit override and as the fallback for an unrecognized `promptVersion` string): v2 generates a fresh `crypto.randomBytes(12).toString('hex')` nonce per call and bakes it into the marker names (`MESSAGES_START_<nonce>`/`MESSAGES_END_<nonce>`, `THREAD_START_<nonce>`/`THREAD_END_<nonce>`) instead of v1's fixed, guessable strings, and serializes the delimited message content as a JSON array of `{username, content}` objects (digest: `{channelName, username, content}`) rather than raw `[username] content` line interpolation — escaping structural characters via `JSON.stringify` instead of relying purely on marker-avoidance, and removing the ambiguity literal newlines/markdown in message content previously introduced around block boundaries. All three build functions (`buildSummaryPrompt`/`buildTaskExtractionPrompt`/`buildDigestPrompt`) now share one generic `build()` that pairs each version with its own `serialize`/`build` step, rather than three near-duplicate functions. `config.js`'s `LLM_SUMMARY_PROMPT_VERSION`/`LLM_TASK_PROMPT_VERSION`/`LLM_DIGEST_PROMPT_VERSION` defaults moved from `v1` to `v2` (an explicit `v1` override remains valid and unaffected) so historical audit rows stay attributable to the prompt version that actually generated them, per this codebase's existing versioning convention; `backend/.env.example` and `RUNBOOK.md`'s env var table updated to match.

A real bug found while writing the disconnect tests, not by inspection: superagent's `Request.abort()` deliberately suppresses that request's own `.end()` callback (its 'error' handler checks `_aborted` and returns without invoking the callback) — a first attempt at these tests `await`ed the aborted request's own promise at the end "to avoid a dangling promise," which instead hung every run for the full Jest per-test timeout, since that promise can now never settle. Fixed by firing the doomed request without ever awaiting its own outcome, only asserting on its *effect* (the queued second request completing, and the concurrency gate's counters).

Tests: `promptTemplates.test.js` gained a "v2 nonce-delimited, JSON-serialized templates" block — the nonce changes on every call and matches `crypto.randomBytes(12).toString('hex')`'s shape; JSON content round-trips markdown/newlines/quotes; a message containing a guessed or copy-pasted marker string (including a fake nonce) cannot forge a boundary, staying entirely inside the real data block between the actual markers; truncation still applies to the serialized content before prompt construction; the existing v1 fixed-marker/fallback tests are untouched and still pass. `aiRoutes.test.js` gained one disconnect-cancellation test per route (summarize, extract-tasks): a mocked provider call that never resolves on its own proves the queued second request can only complete via the first client's disconnect releasing the slot, not by the mock ever settling.

**Verification**: 530 backend tests, 529 passing (the one failure the same pre-existing, previously-documented `aiRoutes.test.js` audit-row race, reproduced identically on a clean `main` via `git stash` before this work began). 92/92 frontend Vitest tests (frontend untouched by either change — both are backend-only). `scripts/verify-audit-log.mjs` run clean (6516 rows, chain intact). ### Serialize DM creation to prevent duplicate-channel races (2026-07-19)

Single Low-severity finding from `docs/reviews/security-performance-review-2026-07-19.md` Finding 9 (`FEATURE_REQUEST.md` entry 3): two people clicking "Message" on each other in the same window (e.g. both opening each other's profile from a shared roster at once) could each pass `POST /api/direct-messages`'s "no existing channel" check under Postgres's default `READ COMMITTED` isolation and create two separate `DIRECT` channels for the same pair, silently breaking the endpoint's "creates or reuses" contract at exactly the concurrency level this app targets.

`POST /api/direct-messages` (`backend/src/routes/directMessages.js`) now takes `SELECT pg_advisory_xact_lock(hashtext(?))`, keyed on the sorted pair of user ids (`dm:{lo}:{hi}`), as the first statement inside its transaction, before the existing-channel check — the same advisory-lock pattern `auditService.js`'s hash-chain append already uses for its own race-prone path (Section 6). Sorting the pair before building the key means both call orderings (a-calls-b, b-calls-a) contend for the identical lock key, so concurrent requests from either direction serialize correctly rather than only the same-direction case being covered. No schema, authorization, or audit change — the lock only affects ordering of an existing check, not who can see or do what, and the existing `CHANNEL_MEMBERSHIP_CHANGE` audit event on channel creation is unaffected.

Tests: `backend/tests/directMessages.test.js` gained a `POST /api/direct-messages concurrency` block using a `fireNow` helper (identical to `aiRoutes.test.js`'s, since supertest/superagent's `Test` is lazy and won't hit the wire until awaited) to fire two requests for the same pair genuinely concurrently, from opposite directions — both resolve to the same channel id, exactly one comes back `201`/`created: true`, and exactly one `DIRECT` channel row exists afterward. A second test confirms existing sequential single-caller reuse behavior is unaffected.

**Verification**: 532 backend tests, 531 passing (the one failure the same pre-existing, previously-documented `aiRoutes.test.js` audit-row race). `scripts/verify-audit-log.mjs` run clean (6522 rows, chain intact). Not independently verified in a live browser this session — the non-racing DM creation path is unchanged and already covered by existing frontend usage, so this rests on the new concurrency test plus the full backend suite. One unrelated, pre-existing issue surfaced while running the full suite this session: after all 532 tests finish, the Jest process itself doesn't exit on its own ("Jest did not exit one second after the test run has completed... asynchronous operations that weren't stopped"), requiring a manual kill — the run's own reported results (531/532, 41.8s) are unaffected, and this matches the same class of latent cross-file leftover-handle fragility documented in the "Hot path splitting" entry above, not a regression from this change.
