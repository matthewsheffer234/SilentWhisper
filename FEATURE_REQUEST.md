# Feature Requests — Running Plan

A living backlog of feature ideas for Silent Whisper. This is a planning document, not an implementation log — see `PROJECT_PLAN.md` Section 11 for what's actually been built.

## How this document is maintained

- Whenever a new feature is requested, a design is thought through (what changes, where, how it fits the existing authorization/audit/rate-limiting conventions in `PROJECT_PLAN.md`) and added below as its own entry, in the format under "Entry format."
- After each addition, every entry is re-ranked by utility and implementation dependency — most useful first unless a prerequisite should land earlier. Rank changes are expected over time as the app's shape changes; nothing here is permanent.
- An entry moves to **Done** (with a pointer to the `PROJECT_PLAN.md` Section 11 log entry that implemented it) once it ships, rather than being deleted — keeps the history of what was considered and why.
- Nothing in this document is authorization to build it. Design-only until explicitly requested.

## Entry format

```
### N. Title

**Status**: Proposed | Planned | Done
**Utility**: one line on why this matters / who it's for
**Origin**: how this came up (a request, a gap found during other work, etc.)

Design: what changes, where, how it's authorized/audited/tested, matching
existing PROJECT_PLAN.md conventions. Enough detail that implementation is
a matter of execution, not re-deciding the approach.
```

### 1. Minimal CI workflow

**Status**: Proposed
**Utility**: `ARCHITECTURE_REVIEW_(Claude).md` calls this "the single cheapest fix on this list" (P0 Recommendation 1) — nothing currently gates a broken build or a failing test from reaching `main`. Confirmed still true: no `.github/workflows` directory exists at all. This is the complementary half of "the deploy loop" entry (already shipped): that entry stops a *stale* container from being deployed; this stops a *broken* one from ever being deployable in the first place.
**Origin**: `ARCHITECTURE_REVIEW_(Claude).md` P0 Recommendation 1 (2026-07-17), re-confirmed against the live repo on 2026-07-18.

Design:
- New `.github/workflows/ci.yml`, triggered on every push to `main` and every pull request targeting it. Two independent jobs so a frontend-only or backend-only PR isn't blocked waiting on the other:
  - **`backend`**: a `postgres` service container (`pgvector/pgvector:pg16`, matching `docker-compose.yml`'s own image choice so `CREATE EXTENSION vector` works — plain `postgres:16-alpine` would fail migration `0009`), `POSTGRES_DB: silent_whisper_test` directly (no separate create-database step needed). Steps: checkout, `actions/setup-node@v4` (node 20, matching every `Dockerfile`'s `FROM node:20-alpine`), `npm ci` in `backend/`, `npm run migrate` (creates the schema *and* the `app_runtime_user` role — see `database/migrations/0007_grants.js`), `npm test`.
  - **`frontend`**: checkout, setup-node, `npm ci` in `frontend/`, `npm run test:unit`, `npm run build`.
  - **Deliberately out of scope**: `frontend/e2e` (Playwright) — needs a live Ollama instance, real Docker Compose stack, cached browser binaries, and careful rate-limiter batching (`RUNBOOK.md`'s Integration Tests section documents exactly why a single unbatched run 429s). That's a meaningfully bigger lift than "run the existing unit/integration suites," and Claude review's own P0 framing only ever asked for `npm test`/`npm run test:unit`/`npm run build` — e2e-in-CI is a separate future entry if wanted, not bundled into this one.
- **Env vars for the `backend` job** (`PGHOST: localhost`, `PGPORT: 5432`, `PGUSER`/`PGPASSWORD` matching the service container's admin credentials, `PGDATABASE: silent_whisper_test`, `APP_DB_USER`, `APP_DB_PASSWORD`, `JWT_SECRET`) come from **GitHub Actions repository secrets** (`secrets.CI_PGPASSWORD`, `secrets.CI_APP_DB_PASSWORD`, `secrets.CI_JWT_SECRET`), not literals in the workflow file — `PROJECT_PLAN.md` Section 3 says exactly this for any future CI secret ("reference secrets only as repository secrets... never embed raw values in workflow YAML or IaC templates") with no carve-out for "it's only a throwaway CI database," so this follows that literally rather than deciding it's a special case. These three repository secrets need to be added once (any generated value works — nothing in the ephemeral CI Postgres container persists or is reachable outside that one workflow run) before the workflow can pass.
- **No schema/authz/audit implications** — this is CI plumbing only, one new file (`.github/workflows/ci.yml`).
- **Tests**: none needed for the workflow file itself; its "test" is that it actually goes green on the PR that introduces it, and red on a PR that deliberately breaks a test (worth confirming once during implementation, then reverting the deliberate break).

### 2. Workspace-scoped Knowledge Explorer for the `[[Entity]]` graph

**Status**: Proposed
**Utility**: The double-bracket entity registry (`[[Entity Name]]` tokenization, `entities`/`message_entities`, entity detail pages) already captures a workspace's implicit knowledge graph as a side effect of normal conversation, but today the only way to reach it is clicking one bracketed mention at a time inside a message. This surfaces it as a first-class, workspace-level destination — what's trending, who actually knows a topic, and where it's been discussed — without ever widening what any individual user is authorized to see.
**Origin**: User feature request (2026-07-20), building directly on the shipped "Double-bracket entity registry, autocomplete, and entity detail pages" entry (`PROJECT_PLAN.md` Section 11, 2026-07-17). Substantially overlaps with backlog entry 4 below ("Entity pages as a lightweight knowledge base") — this entry delivers that vision's discovery/exploration/SME slice now, with a concrete, fully-specified design; entry 4's remaining scope (editable metadata, entity relationships, AI-generated "What we know" summaries) is left there as a distinct follow-on, not duplicated here.

Design:

**Trending Entities** — new `GET /workspaces/:workspaceId/entities/trending?windowDays=&limit=&offset=` in `backend/src/routes/entities.js`, gated `requireAuth` + `requireWorkspaceMember` like every other entity route. Reuses `referencesQuery()`'s exact join shape (`message_entities` → `entities` → `messages` → `channels` → an **INNER JOIN to `channel_members` scoped to `req.user.id`** → `users`) but without an `entity_id` filter, grouped by `(e.id, e.canonical_name)`, `count(*)` (safe here — `message_entities`'s `(message_id, entity_id)` primary key already guarantees at most one row per message per entity, so `count(*)` and `count(distinct message_id)` are identical), ordered `reference_count DESC, e.id ASC` for stable pagination, bounded by `m.created_at >= now() - windowDays` (default 30, env-overridable `KNOWLEDGE_EXPLORER_TRENDING_WINDOW_DAYS`, matching `TASK_DASHBOARD_WINDOW_DAYS`'s existing precedent) and `LIMIT`/`OFFSET`. **Privacy is structural, not a filter bolted on after the fact**: because the join to `channel_members` is an INNER JOIN keyed to the caller, an entity referenced only in a channel the caller isn't a member of contributes zero rows to the `GROUP BY` — it never appears with a `reference_count: 0` placeholder that would itself leak existence, it simply isn't in the result set. This is the same reasoning `GET .../entities/:entityId`'s existing `referenceCount`/`recentReferences` already rely on; trending just runs the ungrouped version of the identical query.

This is `parseOffsetPagination`-shaped (`{entities, total, limit, offset}`), not cursor-based — it's a ranked leaderboard re-sorted by count on every page, not an append-only reverse-chronological feed, matching `parseOffsetPagination`'s documented use case ("bounded... list endpoints browsed in a stable, non-time-based order") rather than the entity-references endpoint's own cursor pattern. `total` is a second `count(distinct e.id)` over the same grouped/joined base. No new migration/index: `channel_members`'s primary key, `idx_messages_channel_date (channel_id, created_at DESC)`, and `message_entities`'s `(message_id, entity_id)` primary key already support this exact join order (start from the caller's small `channel_members` row set → bound `messages` by the date window → join `message_entities` by `message_id`) — confirmed by walking the join order, not assumed; only worth a dedicated composite index if profiling against a large seeded workspace later shows the planner picking a worse order. New `entityTrendingLimiter` (`backend/src/auth/rateLimit.js`, 60 req/60s per user, same shape as `entitySearchLimiter`) — added because this is a workspace-wide aggregation over every visible channel rather than the existing entity routes' single-entity point-reads, making it the first entity route heavier than an indexed lookup.

**Entity Detail View** — extends the existing `EntityDetailsPanel.jsx` / `GET .../entities/:entityId`, no new endpoint. Two changes to what's already there: `entity.description` renders through the existing `renderMessageContent()` (`markdown.jsx`) instead of the current plain `<div>{entity.description}</div>` (`EntityDetailsPanel.jsx:116`) — description is user-authored content, and CLAUDE.md's "treat all user-generated content as untrusted when rendering" applies to it exactly as it does to message bodies; this also delivers the "Description (Markdown)" requirement directly. A new SME section (below Aliases, above the reference count) — see next.

**Subject Matter Experts** — new `GET /workspaces/:workspaceId/entities/:entityId/experts?limit=` in `entities.js`, same `requireAuth`/`requireWorkspaceMember`/`requireWorkspaceEntity` gates as `/references`. Reuses `referencesQuery({workspaceId, entityId, userId: req.user.id})` unmodified — the identical caller-scoped join `/references` and `/:entityId` already use — grouped by `(u.id, u.username, u.display_name)`, `count(*)` as `reference_count`, ordered `reference_count DESC, u.id ASC`, default `limit` 5 / max 20 via `assertBoundedInt`. Same structural privacy guarantee as trending: a user whose only contributions to this entity are in channels the caller can't read simply never appears as a row for that caller — "private discussions must not influence rankings or counts for unauthorized users" falls directly out of the join, it isn't a separate rule to enforce. Response: `[{userId, username, displayName, referenceCount}]`; frontend renders avatar/initials from the existing initials-avatar helper already used for message authors (no new avatar concept), display name first with `@username` shown only when it differs, matching the established "Display names as the primary identity" convention. New `entityExpertsLimiter`, same shape as `entityTrendingLimiter`.

**Citation History** — no new endpoint; `GET .../entities/:entityId/references` (cursor-paginated via `before`, already reverse-chronological) already returns everything the requirement asks for except one field: `serializeReference()` (`entities.js:34-47`) gains `parentMessage: {id, content, username, displayName, createdAt} | null`, populated via a new `LEFT JOIN messages as pm ON pm.id = m.parent_message_id` (plus its own author join) inside `referencesQuery()`, mirroring exactly how `GET /api/search/semantic`'s hits already embed `parentMessage` for the identical reason (see View Context below). `parentMessageId` is already present on every reference row today — the existing field doubles as the thread indicator (`ref.parentMessageId !== null`), no new boolean needed. `snippet`/`timestamp`/`author`/`channel` are already `content`/`createdAt`/`username`+`displayName`/`channelName` on the existing shape.

**View Context (cross-channel navigation)** — a new `onViewContext(ref)` prop threaded from `EntityDetailsPanel.jsx` up through `ChatShell.jsx`, implemented as a thin wrapper reusing `handleNavigateToSearchResult`'s exact shape (`ChatShell.jsx:719-727`): switch workspace if `ref.workspaceId` differs from the selected one, `selectChannel(ref.channelId)`, then `openThread(ref.parentMessage, ref.channelId)` only when `ref.parentMessage` is present. **Explicitly not a new capability**: there is no scroll-to-message-in-the-main-channel-pane anywhere in this codebase today — search-result and mention navigation both resolve a threaded reply to "open the thread sidebar for its root," and resolve a non-reply hit to "switch to the channel," full stop. "View Context" matches this existing precedent exactly rather than inventing a new one; true scroll-and-highlight of an arbitrary top-level message in the main pane is a real, separate gap (shared by search and mentions too, not introduced here) and is out of scope for this entry — worth its own future entry if wanted, not bundled in.

**Frontend entry point** — new `KnowledgeExplorerPanel.jsx` (`Sheet`, wide/tall like `AuditDashboard.jsx`/`BrowseWorkspacesPanel.jsx`), reachable via a new icon-only trigger in `WorkspaceSidebar.jsx`'s per-workspace row (the same treatment "Catch Me Up"'s `Sparkles` trigger already established — a plain `aria-label`-carried icon button, not a new pill) — shows the Trending list; clicking a row opens the existing `EntityDetailsPanel.jsx` (now carrying the SME section and `onViewContext`) exactly as clicking an inline `[[Entity]]` mention already does today via `handleOpenEntity`.

**Authorization note, stated explicitly rather than left implicit**: `GET .../entities/search` and `.../resolve` (used by the composer's `[[Entity` autocomplete) deliberately return entity existence workspace-wide already, regardless of which channels reference the entity — pre-existing, intentional behavior so a user composing a message can discover and reuse a canonical entity name even for a topic they haven't personally seen discussed. This entry does not touch or tighten that; it's called out here only so the difference is a documented decision, not an inconsistency someone finds later and assumes is a bug. The two new aggregation endpoints (trending, experts) do not reproduce that looser behavior — both are built exclusively from the caller-scoped join, never from an unfiltered `entities`/`message_entities` scan with a filtered count layered on top afterward.

**Audit**: none of the four new/changed reads are audited. Matches the existing precedent that `GET .../entities/:entityId`, `/references`, and the task dashboard's `GET /workspaces/:workspaceId/tasks` — all workspace-scoped SQL reads with no LLM cost and no state change — aren't audited today; only routes that spend LLM budget (semantic search, AI summarize/digest) or mutate state get an audit row under this codebase's established convention.

**Tests**: `backend/tests/entities.test.js` gains trending and experts coverage — an entity referenced only in a private channel the caller isn't a member of is absent from that caller's trending results entirely (not present with `reference_count: 0`); a contributor whose only messages on an entity are in such a channel is absent from that caller's SME list; a second workspace member who *is* in the private channel sees both correctly; window/limit/offset bounds and malformed-query 400s; the existing multi-tenancy regression (two workspaces, same entity name) extended to trending/experts. `serializeReference`'s new `parentMessage` field gets references-endpoint tests for both a top-level reference (`parentMessage: null`) and a threaded reply. Frontend: unit coverage for the description-as-Markdown change and the SME section's display-name/`@username` fallback (mirroring `PeoplePicker.jsx`'s `personSecondaryLabel` precedent), plus an e2e test opening the Knowledge Explorer, opening a trending entity, and using View Context to land in the right channel/thread.

### 3. Focus: a finite, responsibility-ranked work-triage view (formerly "Channel attention and health views")

**Status**: Proposed — ground-up redesign of the original entry below; nothing described in this entry is built yet. It is written against the codebase as of 2026-07-24 (the "Ephemeral direct messages" entry is the most recent Done item) and calls out by name every existing piece it reuses, and every place the schema/AI layer has no equivalent today.
**Utility**: A conventional notification/activity panel (this app's own `NotificationPanel.jsx` included) answers "something happened." It does not answer "what, of everything that happened, actually needs *you* — your judgment, your response, your approval, your unblock." In a workspace with real message volume, those are very different lists, and the second one is almost always much shorter. Focus is that second list: the smallest set of conversations, decisions, assignments, approvals, questions, blockers, and knowledge developments that require this specific user's action, ranked by responsibility rather than recency, with an honest, non-gamified "caught up" state when the list is empty.
**Origin**: Original entry requested by the user (2026-07-18), "extending Slack/Teams-style catch-up and activity surfaces into a local, project-focused dashboard." Substantially revised by explicit user direction (2026-07-24) to reframe the feature away from activity-recency ranking and toward responsibility-centric, staleness-aware, momentum-tracked work triage — the product direction recorded in this revision's own request, not inferred.

#### Status

Proposed. No schema, route, service, or UI file for this feature exists in the repository today. This entry is a full specification intended for direct implementation by a later session, phased per "Phased delivery" below. Every reference to an existing file, table, column, or convention in this entry has been checked against current source (see the specific paths cited throughout); every place this design needs something the codebase does not yet have is called out explicitly under "Data and backend considerations" and "Open implementation questions" rather than assumed into existence.

#### Product rationale

Slack-style products optimize, whether they mean to or not, for re-engagement: unread badges, activity feeds sorted by recency, and "since your last visit" framing all reward opening the app and scanning, not resolving anything in particular. That model fits a large, high-recall product where more attention is generically good for the business. It does not fit what this application's own design already prioritizes — a deterministic-first, audit-honest, non-manipulative tool for a project team (`CLAUDE.md`'s rules of engagement, `PROJECT_PLAN.md` Section 3's authorization/audit baseline) — and it actively works against what a project team actually needs, which is fewer stalled things, not more time in the app.

Focus is designed around a different question for every row it shows: not "something happened here," but

1. **What happened?** — a concrete fact (a question was asked, an approval was requested, a checkbox task was assigned, a blocker was confirmed, a decision was reached).
2. **Why does it matter?** — the concrete consequence (it blocks other work, three tasks depend on it, it has been waiting on you for 17 hours).
3. **Why is it being shown to *this* user?** — an authorized, factual reason (you own it, you were asked directly, you follow the entity involved, you participated in the thread) — never an opaque relevance score.
4. **What action would move the work forward?** — a primary action richer than "open the source" (Reply, Approve, Complete, Resolve blocker, Reassign, Acknowledge, and the like — see "Progress and completion model").

The feature explicitly optimizes for reduced manual channel-scanning time, faster responses to explicit requests, fewer stalled conversations, faster blocker resolution, clearer ownership, faster decisions, completed action items, and better cross-channel discovery of developments the user is authorized to see but wouldn't otherwise stumble onto. It explicitly does **not** optimize for notification click-through, messages sent, channel activity, app-open frequency, session duration, or raw unread-count reduction — none of those are tracked as success signals anywhere in this design (see "Metrics"), and the UI section below lists specific patterns (streaks, celebratory effects, oversized cards, vague "AI insight" badges) that are deliberately excluded because they optimize for the wrong thing.

#### Objective

Define a workspace-level **Focus** view that identifies the smallest useful set of items genuinely requiring the current user's judgment, response, or action, ranked by responsibility rather than activity, with explicit staleness, a small explainable momentum model, knowledge-graph-assisted relevance where the graph actually helps, and a real, finite "caught up" state — built entirely on top of this application's existing authorization, audit, rate-limiting, and deterministic-first-then-AI-assisted conventions, never around them.

**On naming**: this entry adopts **Focus** as the working name per the requested product direction. The app's existing vocabulary (`NotificationPanel.jsx`, `mention_notifications`, "Catch Me Up," "Attention" in the original entry) has no strong prior claim on any of "Triage," "Needs You," or "Work Queue" that would override the requested default, so "Focus" is used throughout — `FocusPanel.jsx`, `focus_item_state`, `GET /workspaces/:workspaceId/focus`, etc. A later session is free to bikeshed the literal string before shipping; nothing in this design depends on the name itself.

#### Product principles

- **Responsibility over recency.** Message volume is never, by itself, a ranking input (see "Priority model"). A channel with hundreds of messages since last visit and zero items requiring this user contributes zero Focus rows.
- **Finite and honest.** The view has a real bottom. A user who resolves every explicit obligation sees a calm "caught up" state (see "Progress and completion model"), not backfilled low-value activity to keep the list non-empty.
- **Explainable, not scored.** Every row states its principal reasons in plain language ("why it matters," "why you're seeing this"). An internal numeric/ranking score may exist for ordering purposes, but it is never the explanation shown to the user (Required Design Change 6).
- **Explicit, deterministic, and AI-suggested state are never conflated**, in the data model, the API, or the UI (Required Design Change 7). A confirmed blocker and an AI-guessed possible blocker never look alike.
- **Deterministic first, matching this codebase's own established layering.** Phase 1 and 2 of this feature need no LLM call at all — they are pure SQL over existing/new tables, the same shape as `tasks.js`'s dashboard or `mentionNotificationService.js`'s summary query. AI-assisted detection is Phase 4, gated behind the existing `aiProxyRateLimiter`/`LLM_MAX_CONCURRENT_REQUESTS` machinery, never a prerequisite for the feature being useful.
- **Server time is authoritative for every staleness/momentum computation** (`CLAUDE.md`: "Never trust browser time for anything security- or audit-relevant" — staleness here is not audit-relevant in the forensic sense, but the same principle applies: a client clock must never be able to make an obligation look older or younger than it is).
- **Authorization is structural, not filtered after the fact** — every Focus query is built the same way `entities.js`'s `referencesQuery()`, `tasks.js`'s dashboard query, and `workspaceDigestService.js`'s selection queries already are: an inner join to `channel_members` scoped to the caller, so an unauthorized source contributes zero rows rather than a redacted placeholder (see "Authorization and privacy requirements").
- **Calm, not gamified.** No streaks, no celebratory animation on reaching zero, no color-only urgency signaling (text + accessible status treatment, per Required Design Change 4's own instruction and this app's existing HIG/accessibility pass in `PROJECT_PLAN.md` Section 11).

#### User experience

**Entry point.** `NotificationPanel.jsx` today is a fixed 360px-wide slide-over (`styles.panel`, `width: 360`) built for a flat list of mention rows and membership-invitation alerts — too narrow and too flat for Focus's grouped, multi-field rows (source, staleness, momentum, reasons, entities, primary + secondary action). Focus needs its own wide/tall `Sheet`-based panel, matching the precedent already established by `AuditDashboard.jsx`, `BrowseWorkspacesPanel.jsx`, and backlog entry 2's proposed `KnowledgeExplorerPanel.jsx` — a new `FocusPanel.jsx`. Reachable via a new per-workspace icon-only trigger in `WorkspaceSidebar.jsx`'s workspace row, the same treatment "Catch Me Up"'s `Sparkles` trigger already established (icon-only button, `aria-label` carries the meaning, shown for the currently-selected workspace). `NotificationPanel.jsx` itself is not replaced — mention/invitation alerts stay exactly as they are; Focus is an additional, richer surface layered on top, and Phase 1's "Needs Your Response" category is explicitly allowed to overlap with what the mentions panel already shows (the same underlying `mention_notifications` row can appear in both, since they answer different questions: "was I mentioned" vs. "what does my judgment need to resolve").

**Scope.** Workspace-scoped, matching every other cross-channel surface in this codebase (`tasks.js`'s dashboard, `workspaceDigestService.js`'s digest, entity trending) — all are `GET /workspaces/:workspaceId/...`, gated by `requireWorkspaceMember`. A cross-workspace "everything, everywhere" aggregate is a real, larger idea (this app's own authorization and rate-limiting model is workspace-rooted throughout, so a multi-workspace merge is a materially bigger query-fan-out and privacy-surface question) and is left as an explicit open question, not silently assumed — see "Open implementation questions."

**Layout**, per Required Design Change 9:

- A concise progress summary at the top (see "Progress and completion model" for its exact composition — e.g. "3 of 5 handled · 2 decisions remain," or "You're caught up on high-priority items").
- The count of remaining high-priority obligations, separate from the total row count.
- Grouping by responsibility/state into the sections below, each collapsible, each with its own count.
- Optional filters (by workspace channel, by item type, by momentum state) — client-side over an already-fetched page for v1, no new query surface required.
- A genuine, calm, finite "caught up" state when every section is empty of actionable items — plain text, no celebratory treatment.

**Sections** (Focus categories, detailed below): Needs Your Response · Assigned to You · Waiting or Stalled · Decisions and Outcomes · Knowledge Developments · Suggested by AI · Informational (collapsed by default, visually secondary).

**Row anatomy**, every field required by Design Change 9: source (channel/DM/thread — reuses the channel-name + workspace-name display convention `NotificationPanel.jsx` and `mentionNotificationService.js`'s `mapNotification()` already use), item type, a short context excerpt (rendered through `renderMessageContent()` / `markdown.jsx` like every other piece of user content — `CLAUDE.md`: "treat all user-generated content... as untrusted when rendering"), age/staleness phrase, momentum state where applicable (with its plain-language explanation, never just a badge), "why it matters," "why you're seeing this," affected entities/projects/dependencies (as `[[Entity]]`-style chips linking to `EntityDetailsPanel.jsx`, reusing `handleOpenEntity`), a primary action, and a secondary dismiss/snooze control.

**Explicitly avoided**, per Design Change 9 and this entry's product rationale: an endless chronological feed; ranking primarily by recency; oversized cards; marketing-style summary copy; unexplained red badges; a vague unlabeled "AI insight" pill (every AI-origin row is labeled per "AI and audit requirements"); gamified streaks; celebratory effects on reaching zero.

#### Focus categories

Mapped explicitly to what backs each section, so "Phased delivery" can say precisely which categories exist after which phase:

1. **Needs Your Response** — direct questions addressed to the user, approval/decision requests where the user is the named decision-maker. Phase 1 covers only the deterministic subset reachable via explicit `@mention` + a question-shaped signal or a structured request; broader "likely unanswered question" detection is Phase 4 (AI-assisted, lands in "Suggested by AI" until confirmed).
2. **Assigned to You** — inline checkbox tasks (`- [ ] ... [owner:: @user]`, `services/taskParser.js`) where the parsed `owner` resolves to the current user and the task is unchecked. This is the one category with a real, already-shipped, deterministic backing today (`GET /workspaces/:workspaceId/tasks`).
3. **Waiting or Stalled** — threads/items the user owns, follows, or substantially participated in, where a request is outstanding or replies have stopped (see "Momentum model").
4. **Decisions and Outcomes** — confirmed decisions and their acknowledgement/follow-up state (new concept — no `decisions` table or equivalent exists today; see "Data and backend considerations").
5. **Knowledge Developments** — knowledge-graph-assisted relevance: unusual reference-frequency changes for a followed entity, new relationships, cross-channel co-occurrence (see "Knowledge-graph integration"). Entirely Phase 3.
6. **Suggested by AI** — likely unanswered questions, likely blockers, likely decisions, possible duplicates/contradictions, possible missing owners — always visually and semantically distinct from the confirmed categories above (Design Change 7). Entirely Phase 4.
7. **Informational** — collapsed, visually secondary, everything relevant-but-not-actionable (e.g. a followed entity mentioned without any of the above signals).

#### Priority model

The ranking hierarchy is adopted essentially as specified in the product direction, with an explicit note on which levels this codebase can already back deterministically and which require new state:

1. An overdue assignment owned by the user that blocks other work — **requires new state**: today's inline tasks have no due date and no "blocks" relationship (`taskParser.js`'s `parseTasks()` returns only `{index, checked, text, owner}` — no due date field exists anywhere in `messages.content`'s task syntax or any table).
2. A decision/approval/response explicitly required from the user — **requires a new concept** (no `decisions`/approval-request object exists; see below).
3. An action item assigned to the user — **exists today** via the inline-task `owner` token.
4. A direct question addressed to the user — **partially exists**: `@mention` resolution (`mentionService.js`'s `extractMentionedUserIds`) already identifies "this message was directed at you"; "is this specific message shaped like a question" does not (no question-detection today — Phase 1 can approximate with a mention that ends in `?` or similar deterministic heuristic; true detection is Phase 4 AI-assisted).
5. A confirmed blocker affecting the user's owned work — **requires a new concept** (no blocker/dependency object exists).
6. A thread where someone is explicitly waiting for the user — **requires new state** (see "waiting-on" relationships under "Data and backend considerations").
7. A thread the user owns/follows/participated in that has stalled — **requires momentum computation** (see "Momentum model") plus a definition of "follows/participated" (approximated from `messages.user_id` on a thread's messages — a user has "substantially participated" if they authored a reply in the thread; there is no explicit follow/subscribe concept today, see `conversation_follow` under "Data and backend considerations").
8. A material decision/development involving a followed project/entity/channel — **exists partially** via `entities`/`message_entities` (Phase 3, Knowledge-graph integration); "followed" itself needs a new `entity_follow`/`conversation_follow` concept.
9. General activity in a followed conversation — lowest priority, informational only, never independently promoted by volume.

**Ranking rule, stated explicitly per Required Design Change 1**: recency is a tie-breaker *within* a priority level only. Message volume is never summed into a priority score. A single approval request (level 2) always outranks a hundred-message informational thread (level 9), full stop — this is enforced structurally by grouping into priority buckets first and only ordering by staleness/recency within a bucket, not by any blended score that could let volume compensate for a lower bucket.

#### Staleness model

No universal "N hours" rule (Required Design Change 2). Staleness is computed server-side, per item type, against a configurable response window, and displayed as elapsed-responsibility language ("Waiting on you for 17 hours," "Blocked for 3 days," "No owner after 4 days") rather than a raw timestamp.

**Per-type aging basis**:

| Item type | Ages from |
|---|---|
| Direct question | The message's `created_at` (or the point a response was explicitly requested) |
| Blocker | The point it was confirmed/created |
| Action item (assigned) | Its due date, once due dates exist (see gap above); until then, its assignment time |
| Stalled thread | The point of the last unresolved request with no meaningful reply since |
| Unowned action item | The point it was identified/parsed |
| Pending decision | The point a decision-maker was named |

**Working-time awareness.** `server-generated timestamps only` is already a hard rule in `CLAUDE.md`; what does not exist today is any notion of working hours, working days, or weekends anywhere in the schema or config (`config.js` has no timezone/business-hours concept; `workspaces` has no timezone column — checked against migration `0003_layout_and_hierarchy.js`, whose `workspaces` table is only `(id, name, owner_id, created_at)`). This is a genuine gap, not an oversight to route around: a workspace-level `timezone`/`working_hours` setting (new migration, likely stored the same way other operational values live — either new `workspaces` columns or `app_settings` keys, matching the existing `app_settings`-for-non-secret-config convention from `PROJECT_PLAN.md` Section 4) is required before "17 working hours" / "2 working days" can be computed correctly rather than approximated with wall-clock hours. Until that lands, Phase 2 can ship with a documented wall-clock approximation (explicit in the UI copy, e.g. plain "17 hours" rather than "17 working hours") and upgrade the calculation once the working-time config exists — this sequencing is intentional, not a shortcut: staleness *language* should never overclaim precision the underlying data doesn't support.

**Per-channel/per-item overrides.** `channels` has no `purpose`/`topic`/incident-flag column today (checked against `0003_layout_and_hierarchy.js`: `channels` is `(id, workspace_id, name, type, created_at)`, `type` being only `PUBLIC`/`PRIVATE`/`DIRECT`/`GROUP_DM`). A channel-level response-window override (e.g., a fast incident-response channel needs a much shorter stale-threshold than an async research channel) needs a new nullable column or a `channel_settings`-style table — flagged under "Data and backend considerations." Muted/archived channels: `workspaces`/`organizations` already have `archived_at` (migrations `0008`, `0015`); channels do not have their own archival flag today, but an archived workspace's channels are already excluded from every live query via `requireWorkspaceNotArchived`-style checks, so a Focus item sourced from an archived workspace's channel should be suppressed the same way — no new mechanism needed there, just applying the existing check consistently to Focus's queries.

**Item-level overrides** (an individual decision/blocker explicitly given a longer grace period) are a natural later refinement once the base per-type/per-channel model exists; not required for Phase 2.

#### Progress and completion model

The header summary (Design Change 3) is computed, not decorative: `{highPriorityRemaining, assignedRemaining, waitingCount, decisionsRemaining, caughtUp: boolean}`, derived live from the same query set that produces the rows, never a separately-maintained counter that could drift.

**Distinct states, never conflated in the data model or API** (Required Design Change 8):

- **Read** — the user opened the Focus row's source (a view event, not a state change to the underlying object).
- **Acknowledged** — the user explicitly confirmed awareness (e.g., of a decision) without necessarily completing the underlying work.
- **Dismissed** — removed from *this user's* Focus queue only; writes no shared state. Mirrors `mention_notifications.dismissed_at`'s existing per-recipient-only semantics exactly (`mentionNotificationService.js`'s `dismissed_at` already never affects any other recipient's view).
- **Snoozed** — temporarily hidden until a timestamp or condition, then reappears automatically. Needs a `snoozed_until` (and, later, condition-based snooze) column — no existing snooze concept anywhere in this codebase to reuse.
- **Resolved** — the underlying shared obligation/blocker is complete (shared state — visible to everyone, not just the dismissing user).
- **Completed** — the underlying action item is done. For inline tasks, this already exists: `PATCH /channels/:channelId/messages/:messageId/tasks/:taskIndex` (`messages.js`) flips the checkbox via `setTaskChecked()`, audited as `MESSAGE_TASK_TOGGLED`, broadcasts `message_updated` over the channel's WebSocket room, and re-enqueues the message's embedding job. Focus's "Assigned to You" rows for inline tasks should call this exact existing endpoint as their primary "Complete" action — no new completion mechanism needed for that one item type.

**Opening a row is explicitly never sufficient to resolve an obligation** (Acceptance Criterion 4) — "Read" and "Resolved"/"Completed" are different columns/events, enforced by the state model above, not by UI convention alone.

**No backfill to keep the queue full** (Design Change 3, Acceptance Criterion 8): once a category's query returns zero actionable rows, it renders as empty/caught-up, never populated with lower-priority items to avoid looking finished. This falls out naturally from the priority-bucketed query design (each category is its own bounded query, not a fixed-size page pulled from a merged pool) rather than needing a special "don't backfill" rule bolted on afterward.

#### Momentum model

A small, explainable vocabulary — Moving, Waiting, Stalled, Dormant, Resolved — applied only to bounded work items (a thread with an identified request, a decision, a blocker, an action item), never to a channel's general activity level (Design Change 4's own example: "a busy informational channel may have no momentum state because it does not represent a bounded work item" — this is enforced by only computing momentum for rows that already qualify for one of the "Needs Your Response"/"Assigned to You"/"Waiting or Stalled"/"Decisions" categories, not for "Informational" rows).

| State | Qualifying signals | Disqualifying signals | Transitions | Deterministic or AI-assisted | Example plain-language line |
|---|---|---|---|---|---|
| **Moving** | A substantive reply/update occurred within the item's response window; no unresolved dependency currently blocks the next step | An outstanding request with no reply past its window | From Waiting, once the awaited response/approval lands; from Stalled, once a new substantive reply arrives | Deterministic (reply exists after the request, within window) | "Recent reply from Priya moved this forward." |
| **Waiting** | Progress depends on an identified response/approval/decision/assignment/external event, within its configured window | The window has elapsed (→ Stalled); the dependency is satisfied (→ Moving) | Into Waiting when a request/approval/assignment is created; out when satisfied or the window elapses | Deterministic | "Waiting for Priya for 18 working hours." |
| **Stalled** | A required dependency has exceeded its expected response/completion window | A substantive reply arrives (→ Moving); the item is explicitly resolved (→ Resolved) | From Waiting, once the window elapses; can transition directly to Moving | Deterministic | "This action item is blocking three dependent tasks and is two days overdue." |
| **Dormant** | Activity has ceased with insufficient evidence active work continues (no dependency identified, no recent reply, not explicitly resolved) | Any of: a recent reply, an identified dependency (→ Waiting/Stalled instead), explicit resolution (→ Resolved) | From Moving/Waiting after a long silence with no identified next step | AI-assisted (distinguishing "quietly finished" from "quietly abandoned" needs judgment a deterministic rule can't reliably make — flagged, not guessed at deterministically) | "No activity here in 12 days and no clear next step was identified." |
| **Resolved** | The underlying request/blocker/decision/action is explicitly marked complete | — (terminal) | From any state, only via an explicit Resolve/Complete/Acknowledge action (see "Progress and completion model") | Deterministic (it's a recorded event, not inferred) | "Marked resolved by Priya." |

**Message-frequency independence, stated as a hard rule** (Acceptance Criterion 10): the momentum classifier's inputs are *reply existence relative to a window* and *explicit resolution/dependency events*, never a raw message count. Twenty messages with no decision is still Stalled once the decision-request window elapses; one approval message moves Waiting → Moving regardless of how quiet the thread was before it.

#### Knowledge-graph integration

This app's actual knowledge graph today is `entities` + `message_entities` (migration `0019_entities.js`) plus `[[Entity Name]]` autocomplete/detail pages (`entities.js`) — a workspace-scoped bag of named things and which messages reference them, nothing more. There is **no `entity_relationships` table, no entity metadata beyond `description`/`aliases`, and no "follow an entity" concept** — all three are backlog entry 4's still-`Proposed` scope, not built. This section is explicit about which knowledge-graph signals Focus can build *today* on top of `entities`/`message_entities` alone, and which require entry 4 (or a Focus-specific slice of it) first.

**Buildable today, on the existing `entities`/`message_entities` join shape** (the same authorized join `entities.js`'s `referencesQuery()` already uses):

- Multiple channels discussing the same entity — a `GROUP BY entity_id, channel_id` over `message_entities` joined through the caller's `channel_members`, i.e. exactly `entities.js`'s trending-style query (backlog entry 2) scoped to entities the user has interacted with, restricted to a recent window.
- An entity appearing in a channel the user is authorized to access but doesn't routinely post in — detectable from the same join (the user is a `channel_members` row, so the query surfaces it; whether they "routinely monitor" it is a simple recency-of-the-user's-own-activity-in-that-channel heuristic, not a new concept).
- A new relationship between two entities — **not buildable today**; requires `entity_relationships` (entry 4).
- "Followed entity" unusual reference-frequency increase — **requires a follow concept** (`entity_follow`, new).
- A decision in one channel affecting an action item/project elsewhere, or two discussions describing the same blocker — **requires both the decision/blocker concepts from "Data and backend considerations" and either explicit linking or AI-assisted similarity detection (Phase 4)**; the entity graph alone can suggest "these two threads both mention `[[Project Athena]]`" as a starting signal, but conflating that into "these are the same blocker" is an AI suggestion, never an automatic merge (Design Change 5's own rule: "AI or semantic similarity should not silently merge distinct blockers, decisions, or entities").
- A newly detected expert/owner for an unresolved question — backlog entry 2's proposed SME endpoint (`GET .../entities/:entityId/experts`) is the direct building block once it ships; until then, the same `referencesQuery()`-shaped `GROUP BY user_id` is buildable inline for Focus without waiting on entry 2.
- A stale conversation with a relevant answer elsewhere — **Phase 4, AI-assisted** (needs semantic search over authorized content, i.e. this app's existing pgvector-backed `GET /api/search/semantic`, restricted to the asking user's authorized channels, then surfaced as a "possible" suggestion, never an automatic answer).

**Authorization, restated for this section specifically** because it is the section most likely to leak existence through aggregation if implemented carelessly: every graph traversal Focus performs must use the same inner-join-to-caller's-`channel_members` shape `entities.js`, `tasks.js`, and `workspaceDigestService.js` already establish as this codebase's structural-privacy pattern — an entity referenced only in a channel the caller can't read contributes zero rows to any Focus aggregate, count, or "unusual activity" computation, the same "absent, not zeroed" convention backlog entry 2 already documents for trending/experts. A cross-channel "these two discussions may be related" row must never name or imply the existence of a channel the caller cannot access, even indirectly (e.g. never "also discussed in another channel" as a dangling reference — either the caller can see the linked channel, in which case name it, or the signal is suppressed entirely for that caller).

**Concrete example phrasing**, matching the product-direction examples and this codebase's existing `[[Entity]]` link convention: "Three discussions about `[[Oracle Licensing]]` now reference `[[Project Athena]]`." / "A decision in #architecture may supersede an open action item in #project-athena." — every such row must additionally satisfy the join-based authorization above before it is ever rendered.

#### Channel-health model

Distinct from the per-user Focus queue: a manager-facing view of *coordination quality* for channels the viewer is authorized to see (not a system-admin-only surface — any member with visibility into a channel can see its own health, and a manager role can see health across the channels/workspaces they administer, using the existing `WORKSPACE_ROLE_PERMISSIONS`/`requireWorkspacePermission` model rather than a new permission concept). Indicators are computed from the same underlying signals Focus itself uses (unanswered questions, overdue/unowned action items, unresolved blockers, decisions without linked follow-up, single-participant concentration, duplicate-work suggestions), never from message volume alone.

**Decomposable, not a single score** (Design Change 10's explicit requirement): if an aggregate "channel health" number is shown at all, it must expand into the underlying indicators on click — e.g. "Four direct questions have been unanswered for more than two working days," "Two confirmed decisions have no follow-up owner," "This channel has high message volume but no unresolved assigned work," "A disabled user still owns two open action items."

**A genuinely new gap worth naming explicitly**: "a disabled user still owns two open action items" requires cross-referencing task ownership (`taskParser.js`'s `owner` token, resolved to a `users` row) against `users.is_active`/disabled state — checked, `users` does carry an active/disabled flag already (used by the existing account-disable flow per `PROJECT_PLAN.md`'s WebSocket-eviction-on-disable entry), so this specific indicator is buildable today with no new schema, just a join most existing task queries don't currently perform.

Channel health is Phase 5 in this entry's phased delivery — it depends on the staleness/momentum machinery from Phases 2-3 already existing, and reuses rather than duplicates the decision/blocker/action-item concepts Phase 1-2 introduce.

#### Data and backend considerations

Reviewed against current migrations (`database/migrations/0001` through `0023`) and current services — nothing below assumes a table, route, or service that doesn't already exist unless explicitly marked **new**.

**Reusable as-is:**
- `mention_notifications` (migration `0016`, `read_at`/`dismissed_at` per-recipient) — direct-question/mention detection groundwork for "Needs Your Response."
- `messages.content`'s inline-task syntax + `services/taskParser.js` — "Assigned to You," and its existing completion path (`PATCH .../tasks/:taskIndex`).
- `entities` / `message_entities` (migration `0019`) + `entities.js`'s authorized join shape — "Knowledge Developments" (Phase 3), bounded by what's buildable without `entity_relationships` (see above).
- `message_side_effect_jobs` (migration `0020`) — the `FOR UPDATE SKIP LOCKED` polling-worker pattern (`messageSideEffectsWorker.js`) is the direct template for any new async Focus-item-generation job (e.g. AI-assisted detection in Phase 4), not a new architecture to invent.
- `channel_members`-scoped inner joins — the one and only authorization pattern every new Focus query must reuse (see "Authorization and privacy requirements").
- `app_settings` (migration `0006`) — the existing place for new non-secret operational config (default response windows, default staleness thresholds), following the same env-var-default-plus-DB-override pattern `llm.*`/`dm.autoArchiveDefaultDays` already use.

**New tables/columns this design genuinely requires** (conceptual names per the product direction; final naming should follow this codebase's existing snake_case/singular-table conventions at implementation time):

- **`focus_item_state`** (per-user, per-source-object) — `user_id`, `source_type`, `source_id`, `read_at`, `acknowledged_at`, `dismissed_at`, `snoozed_until`, timestamps. The direct analogue of `mention_notifications`' per-recipient state columns, generalized across source types instead of being mention-specific. This is where "Read/Acknowledged/Dismissed/Snoozed" live — never "Resolved/Completed," which belong on the source object itself (shared state, not per-user).
- **`decision`** (new concept, no existing analogue) — the message/thread a decision was reached in, the named decision-maker(s), `acknowledged_at` per relevant party (or a `decision_acknowledgement` join table, one row per acknowledger), optional linked action items.
- **`work_item_dependency`** (new concept) — represents "blocked by" / "waiting on" relationships between a bounded work item (task, decision, thread) and either another work item or a named person's expected response. This is what backs blocker confirmation, "waiting on you," and "blocks three dependent tasks."
- **`conversation_follow`** / **`entity_follow`** (new concept) — explicit per-user follow state for a thread or entity, since no subscribe/follow mechanism exists anywhere in this codebase today (channel membership is the closest analogue, and it's binary/structural, not a lightweight per-thread follow).
- **Due dates on action items** — today's inline-task syntax (`- [ ] text [owner:: @user]`) has no due-date token. Adding one is a `taskParser.js` regex extension (mirroring exactly how the existing `owner` token was added — same file, same "configurable alias" convention via `config.tasks.*`) plus a fixture-file update (`docs/task-tokenizer-fixtures.json`, which both `backend` and `frontend` test suites read to keep their independent tokenizers in sync — any regex change here must be mirrored in `frontend/src/markdown.jsx` exactly as the file's own header comment already requires for the `owner` token).
- **Channel-level response-window override** — a nullable column on `channels` or a small `channel_focus_settings` table (channel_id, response_window_hours, is_incident_channel, etc.) for Required Design Change 2's channel-purpose-aware staleness.
- **Workspace timezone / working-hours config** — new `workspaces` columns or `app_settings` keys (see "Staleness model" above).
- **AI-origin Focus rows** — need `origin` (`DETERMINISTIC` | `AI_SUGGESTED`), `detector_version`/`prompt_version`, and a `rejected_at`/`rejected_by` column so a user's explicit rejection of an AI suggestion is itself recorded state (Design Change 7 and Acceptance Criterion 14), distinct from a plain dismiss.

**Materialization strategy**: prefer deriving Phase 1-2 Focus rows live from source tables at request time, the same "no second system of record" principle `tasks.js`'s own header comment states explicitly ("a live projection of channel content, not a second system of record"). `decision`/`work_item_dependency`/follows are necessarily persisted (they represent real, explicit facts with no other source of truth), but the *ranked Focus feed itself* should stay a live query over that state wherever query cost allows, exactly like the tasks dashboard, rather than a separately cached/materialized feed that can drift. If a future phase needs caching for cost reasons (e.g. Phase 4's AI-assisted detection, which is expensive enough to justify persisting its output rather than re-running on every page load), invalidation must cover: new messages, replies, assignments, completions, decisions, blocker resolution, entity merges (once entry 4 exists), deleted content (no message-deletion feature exists today, per `messageSideEffectsWorker.js`'s own defensive comment, so this is forward-looking), channel membership changes, permission revocation, and user deactivation — mirroring the same "a user losing access must also lose access to any cached/derived data" rule this app already enforces structurally for mentions and digests.

#### Authorization and privacy requirements

Every one of the following is derived only from content the caller is currently authorized to access, using the established inner-join-to-`channel_members` structural pattern, not a filter applied after a broader query:

- Every Focus row, category count, progress count, health indicator, staleness calculation, momentum state, entity connection, cross-channel insight, AI input/output, explanation, and any graph traversal.
- Unauthorized private content must not leak through aggregate counts, hidden-source placeholders, inferred relationships, entity activity totals, timing side-channels, dependency counts, momentum summaries, "another channel" language, AI-generated summaries, or error behavior — matching `PROJECT_PLAN.md` Section 3's existing existence-hiding convention (not-a-member → 404, identical to a genuinely nonexistent resource) applied consistently to every new Focus surface.
- A private channel's existence must never leak through an inferred cross-channel relationship (Knowledge-graph integration's own rule above).
- A user losing access to a source (channel membership removed, workspace archived, account disabled) must also lose access to any cached/persisted Focus item derived from that source — enforced the same way `requireWorkspaceMember`'s live re-check already governs every other cross-channel query in this codebase, not by a separate cleanup job.
- System-admin structural access (`requireWorkspaceMemberOrSystemAdmin`, `membershipService.js`) must **not** implicitly grant Focus visibility into message content the admin isn't a genuine `channel_members` row for — the same explicit boundary `membershipService.js`'s own doc comments already draw ("Deliberately does NOT extend to message content... an admin who administers a private workspace this way still cannot read a single message in it"). Focus is a message-content-derived surface, so it follows the member-only path, never the structural-management override.

#### AI and audit requirements

- **Ordinary per-user Focus state — read, dismiss, snooze, local sort/filter — does not create high-volume audit rows**, matching the existing precedent that `mention_notifications.read_at`/`dismissed_at` writes, `GET .../entities/:entityId`, and the task dashboard's reads are unaudited today (pure per-user UI state, no content exposure beyond what the caller could already see, no LLM cost).
- **Shared-state changes follow the audit policy of the underlying object**: completing/reassigning an action item already writes `MESSAGE_TASK_TOGGLED` (existing); a new decision-acknowledgement or blocker-resolution action should get its own audit event type in the same style (`DECISION_ACKNOWLEDGED`, `BLOCKER_RESOLVED`, `WORK_ITEM_DEPENDENCY_CREATED`/`_RESOLVED`), following `appendAuditEvent()`'s existing shape (`actorId`, `actorIp`, `actionType`, `targetResource`, ids/counts-only `payload` — never raw message content, per `auditService.js`'s own established convention).
- **AI-assisted detection (Phase 4) reuses the existing provider abstraction end to end** — `runStreamingCompletion`, `buildXPrompt`-style prompt builders in `llm/promptTemplates.js`, `aiProxyRateLimiter`/`LLM_MAX_CONCURRENT_REQUESTS`, and the audit shape `AI_SUMMARIZE_REQUESTED`/`AI_TASK_EXTRACTION_REQUESTED`/`AI_WORKSPACE_DIGEST_REQUESTED` already establish (`provider`, `promptVersion`, `truncatedInputLength`, `wasTruncated`, source/generated item counts — never raw private content). **A real gap worth naming**: every existing AI route streams free-text/markdown back to the client and persists nothing structured (`ai.js`'s `extract-tasks` and `workspace-digest` routes are pure request/response, no stored output). Phase 4 needs the *opposite* shape — structured, individually addressable suggestion rows a user can accept/reject one at a time — so it needs either a new structured-output prompt convention (e.g. requesting JSON and parsing it server-side, with the same untrusted-output handling `CLAUDE.md` already requires for any LLM output rendered in the UI) or a per-item follow-up call pattern. This is flagged as new design work within Phase 4, not assumed solvable by reusing the streaming pattern unchanged.
- **AI suggestions are never visually or semantically equivalent to confirmed state** (Design Change 7/13, Acceptance Criterion 13) — distinct section ("Suggested by AI"), distinct `origin` column (see "Data and backend considerations"), plain-language confidence labels only ("Possible"/"Likely"/"Needs review"), never a numeric confidence percentage unless the underlying model is calibrated and that calibration has been validated (Design Change 7's explicit instruction) — this app's Ollama/vLLM adapters are not calibrated classifiers today, so numeric confidence is out of scope unless a future session does that validation work first.
- **Users can reject AI/graph-derived suggestions** (Acceptance Criterion 14) — a rejection is itself recorded (`rejected_at`/`rejected_by` on the Focus item state), and a rejected suggestion should not resurface identically on the next detection pass (the detector needs to consult prior rejections the same way it would consult prior dismissals, or risk becoming its own annoyance).

#### Metrics

Deliberately the inverse of a typical engagement dashboard, matching the product rationale above:

- Time from an item becoming stale to its resolution (should trend down).
- Count of items resolved via their primary action vs. dismissed-without-action (a high dismiss rate on a category is a signal that category's detection is miscalibrated, not that users are disengaged).
- Fraction of "caught up" states reached per user per day (a *higher* number here is success — the opposite of a typical unread-count/session-duration metric).
- AI suggestion acceptance/rejection rate, tracked per detector/prompt version (feeds directly into Phase 4's "precision and false-positive evaluation," Phase 4's own stated goal).
- **Explicitly not tracked as a success metric**: Focus panel open count, time spent in the panel, or any click-through rate — consistent with "what it should not optimize for" in the product rationale. These may still exist as basic operational telemetry (the same way any route naturally has request logs) but must never be treated as a product-health signal for this feature.

#### Phased delivery

**Phase 1 — Explicit Focus Queue** (no new AI, minimal new schema): unread direct mentions (reuses `mention_notifications` as-is); a deterministic question heuristic (explicit `@mention` + simple question-shaped signal) for "Needs Your Response"; assigned action items (reuses `taskParser.js`/`GET .../tasks` as-is); replies in threads the user authored/replied in (derivable from existing `messages.parent_message_id` + `messages.user_id`, no new table); new `focus_item_state` for per-user read/dismiss/snooze/acknowledge/complete distinctions; responsibility-based ordering per the priority model (bucketed, not a single blended score); a real finite caught-up state; "why you're seeing this" reasons drawn only from facts already computed above.

**Phase 2 — Staleness and Momentum**: configurable response windows (`app_settings`-backed defaults, per the existing config convention); working-time calculations (blocked on the workspace-timezone/working-hours gap identified above — ship with an explicit wall-clock approximation first if needed, upgrade once that config exists); overdue/stalled states; new `work_item_dependency` for explicit blocker/waiting-on relationships; due dates on inline tasks (`taskParser.js` extension + fixture sync); momentum transitions per the table above; duplicate suppression (a deterministic near-duplicate check, e.g. same entity + same channel within a short window, ahead of any AI-assisted duplicate detection in Phase 4); progress summaries.

**Phase 3 — Knowledge-Graph Focus**: `entity_follow`/`conversation_follow`; authorized cross-channel entity developments and related-discussion detection, bounded by what's buildable on `entities`/`message_entities` alone per "Knowledge-graph integration" above; linked decisions/actions (depends on Phase 2's `decision`/`work_item_dependency`); possible-duplicate-blocker surfacing (as an AI-labeled suggestion once Phase 4 lands, or omitted until then rather than guessed at deterministically); relevant-expertise discovery (reuses backlog entry 2's SME query shape if entry 2 has shipped by then, otherwise the equivalent inline query); clear graph-derived explanations, authorization-checked per-row as specified above.

**Phase 4 — AI-Assisted Triage**: likely-unanswered-question / likely-blocker / likely-decision / possible-missing-owner / possible-contradiction / possible-duplicate-work detection; requires the new structured-output AI response shape flagged under "AI and audit requirements" (a real, non-trivial addition to `llm/promptTemplates.js`/`llm/aiService.js`, not a reuse of the existing streaming-markdown pattern unchanged); user confirm/correct/reject flows; precision and false-positive tracking per "Metrics."

**Phase 5 — Channel Health**: authorized, decomposable team-level health indicators per the "Channel-health model" above; dependency/ownership analysis (including the disabled-user-still-owns-work indicator, buildable today); repeated-question and abandoned-request detection (AI-assisted, reuses Phase 4's detection infrastructure); high-volume/low-outcome detection; manager views scoped to channels the viewer already has permission over via `WORKSPACE_ROLE_PERMISSIONS`, no new permission model; trends based on coordination outcomes, never message volume.

#### Testing requirements

Extending this codebase's existing test conventions (`backend/tests/*.test.js`, negative-authorization tests per `CLAUDE.md`, the private-channel-filtering pattern already established in `entities.test.js`/`directMessages.test.js`):

- Workspace/user isolation and private-channel filtering for every new Focus query, following the exact "absent, not zeroed" assertion style backlog entry 2's trending/experts tests already use.
- Existence-hiding: a workspace/channel/entity/decision the caller isn't authorized for behaves identically whether it exists or not, across every new endpoint.
- DM/thread isolation consistent with `directMessages.js`'s `workspace_id IS NULL` exclusion precedent, where relevant.
- Read vs. acknowledged vs. resolved vs. completed vs. dismissed vs. snoozed are asserted as genuinely distinct state transitions, not aliases of each other.
- Deterministic Focus-item generation is reproducible from fixed source data (same "pure function over known input" testing style `taskParser.js`'s own tests already use).
- Responsibility-based ranking: a seeded scenario with one high-priority item and many low-priority/high-volume items asserts the high-priority item ranks first (directly testing Acceptance Criterion 1 and 10).
- Staleness calculations, including working-hours/weekend/timezone behavior once that config exists, and the documented wall-clock-approximation behavior before it does.
- Channel-specific response-window overrides.
- Momentum-state transitions, including the "message volume alone cannot cause Moving/Stalled" rule as a dedicated regression test (a scenario with twenty messages and no decision must classify as Stalled, not Moving).
- Duplicate suppression.
- Snooze-and-resurface timing.
- Dismissal confirmed not to modify any shared/underlying object state (only `focus_item_state`).
- Completion/blocker-resolution correctly updates or removes the corresponding Focus row.
- Source deletion and immediate access-revocation both correctly hide any derived/cached Focus item (once caching exists, per the invalidation list above).
- Entity merge/split effects on Focus rows (once backlog entry 4's `entity_relationships`/merge concept exists).
- Graph-derived relevance filtering, with a dedicated cross-channel-privacy-leakage regression test mirroring entry 2's own multi-tenancy test.
- AI suggestion labeling, confirm/reject flows, and false-positive correction, once Phase 4 exists.
- Progress-summary accuracy against seeded fixtures.
- Genuine caught-up behavior: a seeded user with zero actionable items sees the calm caught-up state, and it is never backfilled.
- Frontend/e2e: opening a Focus row navigates to the correct channel/thread/entity/decision/action item, mirroring `handleNavigateToSearchResult`'s existing navigation pattern.

#### Acceptance criteria

1. A user with hundreds of unrelated unread messages but one assigned approval sees the approval ranked above general activity.
2. Every Focus row states why it matters and why the user is seeing it, in plain language, not a numeric score.
3. A direct request becomes stale according to a configured (initially wall-clock, later working-time-aware) response window, never a single hardcoded universal threshold.
4. Opening a row does not, by itself, mark the underlying obligation resolved or completed.
5. Completing the underlying action (e.g. via the existing `PATCH .../tasks/:taskIndex`) removes or updates the corresponding Focus row.
6. Snoozed items return automatically at the correct time.
7. A user can clear every explicit obligation and reach a genuine, calm caught-up state.
8. Low-value informational activity is never inserted to prevent an empty queue.
9. Momentum can transition from Waiting to Moving once the required dependency is satisfied.
10. Message volume alone never causes an item to be classified as Moving, Stalled, urgent, or high-priority.
11. A graph-derived row names the specific entity/relationship responsible for its relevance.
12. No inaccessible channel contributes to any visible row, count, momentum state, health indicator, entity trend, or AI output, for any caller.
13. AI suggestions are visually and semantically distinct from confirmed assignments, blockers, and decisions at all times.
14. A user can reject an incorrect AI or graph-derived suggestion, and that rejection is recorded and respected on future detection passes.
15. Channel health presents specific, decomposable underlying conditions, never a bare unexplained score.

### 5. Admin Analytics Dashboard — collaboration structure and interaction trend

**Status**: Proposed
**Utility**: Shows admins *structural* collaboration patterns — who bridges across otherwise-siloed channels/workspaces, and how much cross-conversation interaction is actually happening over time — without ever reading a message's content. Useful for spotting silos, single points of failure, and teams that have stopped talking to each other.
**Origin**: Same discussion as the Admin Analytics Dashboard's Activity entry (2026-07-24), the "collaboration graph" idea from that conversation. Depends on that entry's dashboard shell and shares its rate limiter and DM-exclusion rule — see "Done" below (`AdminAnalyticsPanel.jsx`, `adminAnalyticsLimiter`, `backend/src/routes/adminAnalytics.js`).

Design: two distinct signals, kept separate rather than merged into one metric, because they answer different questions and only one of them can be trended over time with the schema as it exists today.

- **Membership-overlap graph (structural, snapshot only, zero `messages` access)** — `channel_members` has no `joined_at`/`created_at` column (checked: migration `0003_layout_and_hierarchy.js`'s `channel_members` is just `(channel_id, user_id)` primary key, nothing else), so this signal is a current snapshot, not a trend — stated here explicitly as a known limitation rather than silently only showing half of what "over time" implies; adding `joined_at` in a future migration is the natural way to make this trendable later, out of scope for this entry. `GET /api/admin/analytics/collaboration/membership-graph?scope=organization|workspace&scopeId=&minSharedChannels=` — self-join `channel_members cm1 JOIN channel_members cm2 ON cm1.channel_id = cm2.channel_id AND cm1.user_id < cm2.user_id` (the `<` avoids double-counting and self-pairs), joined through `channels` (`workspace_id IS NOT NULL`, plus the scope filter), grouped by the user pair, `count(distinct cm1.channel_id) as sharedChannels`. `minSharedChannels` (`assertBoundedInt`, default from new `ADMIN_ANALYTICS_MIN_SHARED_CHANNELS` env var, default `2`) suppresses pairs at or below the threshold before they're returned — a deliberate anti-inference control, the same spirit as entry 2's structural privacy: a pair sharing exactly one channel is common and not a meaningful "bridge" signal, and returning it invites over-reading a single coincidence as a relationship. Response: `{nodes: [{userId, username, displayName}], edges: [{userA, userB, sharedChannels}]}`.
- **Interaction trend (reply-based, time-bucketed, still metadata only)** — uses `messages.parent_message_id` joined back to the parent's `user_id` (who replied to whom), never `content`. `GET /api/admin/analytics/collaboration/interaction-trend?scope=organization|workspace|channel&scopeId=&windowDays=&bucket=day|week` returns `{buckets: [{bucket, replyCount, distinctPairCount}]}` — an aggregate volume/breadth trend, not a per-pair-per-bucket edge list (a full pairwise breakdown trended per bucket would grow unboundedly with user count × bucket count for little added insight over the snapshot graph above, so the time dimension is deliberately kept to aggregate counts only; the *structure* of who-talks-to-whom stays in the snapshot endpoint). A `minInteractions` suppression threshold (`ADMIN_ANALYTICS_MIN_INTERACTIONS`, default `3`) applies if a future revision adds a per-pair breakdown here — noted now so the convention is established before that data shape exists, not retrofitted.
- Authorization/scope: identical `requireSystemAdmin` gate and DM exclusion as the shipped Activity endpoints; both endpoints on `adminAnalytics.js`, sharing `adminAnalyticsLimiter`.
- No new migration — both queries run entirely over `channel_members`'s existing primary key (already indexes `channel_id` as its leading column, so the self-join needs no new index) and `messages`'s existing `parent_message_id`/`user_id` columns; revisit only if profiling against a large seeded workspace shows a worse plan, matching entry 2's index-reasoning precedent.
- Audit: none, same reasoning as the shipped Activity entry — pure aggregate reads, no content.
- Frontend: second `AdminAnalyticsPanel.jsx` tab, "Collaboration," added to that component's existing `tabs` array (today a single-entry array with only "Activity"). No graph-visualization library exists in this frontend; rather than pull one in for v1, the membership graph renders as a sortable table of user pairs + shared-channel count (same "table first" reasoning as the Activity tab), and the interaction trend renders with the same bucket-table-plus-CSS-bar treatment. A real node-link visualization is an explicit, non-blocking follow-on once the data shape has been validated against real usage.
- Tests: membership-overlap counts only non-DM shared channels; a pair at or below `minSharedChannels` is absent from the response entirely (not present with a suppressed value — same "absent, not zeroed" convention entry 2's trending endpoint established); interaction-trend bucketing correctness and DM exclusion; malformed params 400; non-system-admin 403.

### 6. Admin Analytics Dashboard — aggregate semantic/sentiment trend

**Status**: Proposed
**Utility**: Directly answers the originating question — can admins see semantic/sentiment trends per user, organization, workspace, and channel over time without exposing private content. Answer: yes, by reusing the embedding this app already computes for every message (semantic search, entry 1) rather than adding a second per-message LLM call.
**Origin**: Same discussion as the Admin Analytics Dashboard's Activity entry and entry 5 (Collaboration) above (2026-07-24) — this is that conversation's actual "semantic score per message" proposal, scoped down to an aggregate, privacy-conscious v1. Depends on the shipped Activity entry's dashboard shell and shares its rate limiter and DM-exclusion rule.

Design:
- **Reuses the existing embedding, computed once.** `search/embeddingWorker.js`'s `processJob()` already calls `embedText()` once per message and holds the resulting vector in memory before writing it to `message_embeddings`. This entry extends that same function with one more step in the same transaction: compute a tone score as `cosineSimilarity(embedding, positiveAnchorEmbedding) - cosineSimilarity(embedding, negativeAnchorEmbedding)` (pgvector's `<=>` is cosine *distance*; `1 - distance` gives similarity, same conversion `search.js`'s `similarity` field already uses) and insert it into a new `message_sentiment_scores` table. This deliberately does not call the LLM adapter a second time or add a second queue/worker — the marginal cost per message is pure arithmetic on a vector already in hand, not a new network round trip to Ollama. Anchor phrases (`SENTIMENT_POSITIVE_ANCHORS`/`SENTIMENT_NEGATIVE_ANCHORS`, comma-separated env vars with sane built-in defaults, e.g. "thanks, great work, appreciate it, sounds good" / "frustrated, blocked, this is broken, not working") are embedded once, lazily, on first use and cached in-process for the life of the process — a model change already requires a restart (`config.embedding.dimension`'s existing doc comment says as much), so no cache-invalidation logic is needed beyond that.
- **This is a coarse proxy, stated as such everywhere it's surfaced.** Embedding-similarity-to-a-fixed-anchor is a known cheap-but-approximate stand-in for real sentiment classification — good for a directional trend line at aggregate scale, not a calibrated per-message judgment. This caveat belongs in the UI itself (see Frontend below), not only in this design doc, given how easy it would be for an admin to over-read a noisy score as a fact about a person.
- New migration `0025_message_sentiment_scores.js`: `message_sentiment_scores (message_id UUID PRIMARY KEY REFERENCES messages(id) ON DELETE CASCADE, score REAL NOT NULL, model TEXT NOT NULL, computed_at TIMESTAMPTZ DEFAULT NOW())` — same one-row-per-message shape as `message_embeddings`, no separate index needed since the trend query joins through `messages.created_at` for bucketing, not this table's own timestamp.
- `GET /api/admin/analytics/sentiment-trend?scope=organization|workspace|channel|user&scopeId=&windowDays=&bucket=day|week` — joined through `channels` (`workspace_id IS NOT NULL`, same DM exclusion as the Activity/Collaboration entries) and, for `user` scope, filtered on `messages.user_id`. Grouped by `date_trunc(bucket, m.created_at)`, `avg(score) as avgScore`, `count(*) as messageCount`. A bucket with fewer than `SENTIMENT_MIN_BUCKET_MESSAGES` (default `5`) messages is dropped from the response entirely rather than returned with a wide-variance average — the same small-N-average problem a single terse or sarcastic message could otherwise dominate, called out explicitly as an anti-overinterpretation control (same spirit as entry 5's edge-suppression threshold).
- **`scope=user` is a sharper privacy tradeoff than the other three and is treated differently, not just permitted.** Organization/workspace/channel scores are inherently many-person averages; a `user` score is individual tone-monitoring, which is the closest this feature gets to surveillance of a specific person. It stays in-scope here because the original question explicitly asked for per-user trends and there's a real "is this person burning out / disengaging" use case for it — but `POST`-equivalent visibility into it is logged: a new `AI_SENTIMENT_TREND_VIEWED` audit event is written (`actorId`, `targetUserId`, `windowDays`, `bucket`) whenever `scope=user`, and only then — organization/workspace/channel-scoped views stay unaudited, matching the Activity/Collaboration entries' "pure aggregate read" precedent, since only the individual-level case rises to the same "worth knowing this was looked at" bar `AuditDashboard.jsx`'s own content-exception logging already uses. Rate-limited under the same `adminAnalyticsLimiter` as the other tabs.
- Audit event added to `audit/auditService.js`'s known action-type set alongside the existing `AI_*` events.
- Frontend: third `AdminAnalyticsPanel.jsx` tab, "Sentiment Trends," same bucket-table-plus-CSS-bar rendering as the Activity/Collaboration tabs, scope selector defaulting to `workspace` (not `user`) on open. A persistent, non-dismissable caveat line ("Approximate tone signal derived from message embeddings — not a clinical or HR measurement") is always visible on this tab, and becomes more prominent (e.g. a distinct banner, not just a footnote) when `scope=user` is selected, since that's the one variant where over-reading a noisy number as a fact about a specific person is the real risk.
- Tests: `backend/tests/adminAnalytics.test.js` (extending the shipped Activity/entry-5-Collaboration suite) — anchor-similarity scoring is deterministic and unit-testable directly given fixed embeddings (mock the adapter, assert the score formula); the embedding-worker extension writes both `message_embeddings` and `message_sentiment_scores` from a single `embedText()` call, not two (assert via a call-count spy on the adapter); trend bucketing and the minimum-bucket-size suppression; DM exclusion; `scope=user` writes exactly one `AI_SENTIMENT_TREND_VIEWED` audit row per request while `scope=workspace`/`organization`/`channel` write none; non-system-admin 403; `scripts/verify-audit-log.mjs` stays clean after the new audit event type is added.

### Fix the `aiRoutes.test.js` audit-row race at its root

**Status**: Done — see `PROJECT_PLAN.md` Section 11, "Fix the aiRoutes.test.js audit-row race at its root" (2026-07-20). Originally ranked entry 2.

Root cause: `backend/src/llm/aiService.js`'s `runStreamingCompletion()` called `res.write(text)`/`res.end()` itself on the success path — finishing the HTTP response — and only *then* returned control to the calling route, which did `await appendAuditEvent(...)` afterward. ### Bounded AI concurrency queue instead of hard-rejecting

**Status**: Done — see `PROJECT_PLAN.md` Section 11, "Bounded AI concurrency queue instead of hard-rejecting" (2026-07-18).

`llm/concurrencyGate.js`'s synchronous `tryAcquire`/`release` pair became an async `acquireSlot(maxConcurrent, { onQueued })`/`release()`, backed by a bounded in-memory FIFO queue (`AI_QUEUE_MAX_DEPTH`, default 8, env-only). A request beyond `LLM_MAX_CONCURRENT_REQUESTS` now waits rather than being refused outright; `release()` transfers the freed slot directly to the next queued waiter instead of decrementing and letting a fresh caller race for it. ### Hot path splitting: async notification writes and entity linking

**Status**: Done — see `PROJECT_PLAN.md` Section 11, "Hot path splitting: async notification writes and entity linking" (2026-07-18). Before implementation, confirmed with the user that the resulting ~1s lag on mention notifications was acceptable, given the design's own text flagged this as a UX tradeoff needing direct sign-off rather than an assumption — both notification writing and entity linking moved to the worker, per that confirmation, rather than the "notifications stay sync" fallback also offered.

New migration `0020_message_side_effect_jobs.js`: `message_side_effect_jobs` (`message_id`, `job_type` — `NOTIFICATION` | `ENTITY_LINK` — composite primary key, `status`, `attempts`, `last_error`, timestamps), same `FOR UPDATE SKIP LOCKED`-polling shape as `embedding_jobs`, same grants. `routes/messages.js` and `ws/server.js`'s send paths both replaced their inline `extractMentionedUserIds`/`createMentionNotifications`/`linkMessageEntities`/mention-WS-push calls with a single `enqueueMessageSideEffectJobs(db, { messageId, workspaceId })` (new `services/messageSideEffectsQueue.js`) — a `NOTIFICATION` row is always enqueued, an `ENTITY_LINK` row only when `workspaceId` is truthy (DIRECT/GROUP_DM channels never had entity linking to begin with). New `workers/messageSideEffectsWorker.js`, structurally identical to `embeddingWorker.js`: polls on `MESSAGE_SIDE_EFFECTS_WORKER_INTERVAL_MS` (default 1s — tighter than embedding's 2s, since this feeds a user-visible notification rather than background search indexing), re-derives everything a job needs (message content, channel, workspace, sender identity) by joining `messages`/`channels`/`users` at process time rather than carrying payload columns, and — since the whole mention pipeline moved, not just the DB write — the real-time `mention` WS push (via `sendToUser`) now happens inside the worker's `NOTIFICATION` job processing, not synchronously at send time.

Three deliberate deviations from the original design text, each because the actual codebase precedent it was mirroring turned out to work differently than the design assumed:
- **No `processed_at` column, delete-on-success instead**: the design listed `processed_at` alongside `status`/`attempts`, but `embedding_jobs` — the explicitly-named template — has no such column and deletes its row on success, keeping the table free of an ever-growing "already processed" backlog. Followed `embedding_jobs`'s actual shape over the design text's `processed_at` mention.
- **No same-transaction enqueue**: the design called for the job-enqueue insert to share a transaction with the message insert. `enqueueEmbeddingJob` — the identical problem, already shipped — does not do this; it's a separate, try/catch-wrapped, best-effort insert immediately after, accepting a narrow crash window as a documented rare gap rather than a data-loss risk. Matched that precedent instead of introducing a new transactional pattern.
- **Retry/dead-letter tested via an intentionally unrecognized `job_type`, not a mocked failure**: unlike embedding jobs (which call an external provider, mockable via `global.fetch`), these jobs are pure DB work with nothing external to mock. `processJob` throws on any `job_type` other than `NOTIFICATION`/`ENTITY_LINK` — a real, deterministic error path (also useful defensively against future bad data) — and the tests insert a `'BOGUS'`-typed row directly to exercise retry/dead-letter without mocking anything.

### WebSocket connection hygiene: immediate eviction on account disable + payload cap

**Status**: Done — see `PROJECT_PLAN.md` Section 11, "WebSocket connection hygiene: immediate eviction on account disable + payload cap" (2026-07-17).

Both independent architecture reviews' top P0 item. New `disconnectUser(userId, { code, reason })` in `backend/src/ws/connectionRegistry.js` force-closes every live socket for a user (default close code `4004`, reason `"Account disabled"`, an error frame sent first since a close `reason` string isn't reliably surfaced to browser client code); `backend/src/routes/admin.js`'s `POST /users/:userId/disable` calls it synchronously right after `revokeAllRefreshTokensForUser`, in the same request as the `status: 'DISABLED'` update — not queued, since "immediately" was the whole point. That alone only handles sockets already open, so `requireAuth` (`backend/src/auth/requireAuth.js`) now re-checks `users.status === 'ACTIVE'` against the database on every REST call (previously JWT-verification-only, trusting whatever was true at token-issue time), and `ws/server.js`'s `handleAuthenticate` mirrors the identical check for both a first authenticate frame and a re-authenticate frame on an already-open socket — both paths reject with the same generic "Invalid or expired access token" message a truly invalid/expired token gets, so a disabled account can't be distinguished from either case. Separately, `new WebSocketServer(...)` gained `maxPayload: config.ws.maxPayloadBytes` (new `WS_MAX_PAYLOAD_BYTES`, default 131072/128 KiB — sized above the worst-case UTF-8 encoding of a `MAX_MESSAGE_LENGTH` message plus JSON framing overhead, so no legitimate frame is ever at risk), closing an unauthenticated memory/CPU exhaustion vector before a frame is ever buffered or reaches `JSON.parse`.

A real bug found by testing, not by inspection: exceeding `maxPayload` surfaces as an `'error'` event on the per-connection socket (`ws`'s receiver rejects the frame before `'message'` ever fires), and `wss.on('connection', ...)` had no `'error'` listener on individual sockets anywhere — an EventEmitter `'error'` event with no listener is a fatal, process-crashing exception in Node, not just a dropped event. Fixed with a no-op `ws.on('error', () => {})` per connection; `ws` already closes the socket itself (code `1009`) once it emits this, so nothing else was needed. Also updated `RUNBOOK.md`'s WebSocket Protocol close-code table with `4004` and the new payload cap.

**Tests**: `backend/tests/ws.test.js` — disabling a connected user force-closes their socket within the same test (not just on a later reconnect attempt); a disabled user's still-unexpired token is rejected on a fresh authenticate attempt; a disabled user's still-unexpired token is rejected on a re-authenticate frame sent over an already-open socket (status flipped directly via the DB in this one, isolating the reauth branch from the separate eviction path the first test covers); a frame larger than `WS_MAX_PAYLOAD_BYTES` closes the connection with code `1009`. `backend/tests/auth.test.js` gained a test proving a still-unexpired access token is rejected by `GET /api/auth/me` on the very next request after disable (previously only login/refresh were covered). 436 backend tests (435 passing, the one failure the same pre-existing `aiRoutes.test.js` audit-row race documented in prior entries — reproduced identically on a clean `main` via `git stash`, confirming it predates this change).

`pgvector/pgvector:pg16` replaces `postgres:16-alpine` (migration `0009_pgvector_and_embeddings.js`: `vector` extension, `message_embeddings` + HNSW cosine index, `embedding_jobs` work-queue table). `llm/adapters/*.js` gained `embed()` alongside `generate()`/`checkHealth()` for all three providers, reusing the live `app_settings`-overridable provider rather than a separate config surface. New `backend/src/search/` (`embeddingConcurrencyGate.js`, `embeddingService.js`, `embeddingQueue.js`, `embeddingWorker.js`) handles async, failure-tolerant ingestion — enqueued as a sibling step at both the REST and WS message-send call sites, processed by a polling worker with retry/dead-letter. `POST /api/search/semantic` (`routes/search.js`) embeds the query, ranks by cosine similarity, and authorizes via the existing membership helpers plus a `channel_members`-filtered join for the cross-channel case; audited as `AI_SEMANTIC_SEARCH_REQUESTED` with query length/model/result count, never raw query text. Frontend: `SemanticSearchPanel.jsx`, reachable via a "Search" button available to every user. 196/197 backend tests (one pre-existing, unrelated flake), 1 new e2e test passing in isolation.

### Admin dashboard: user provisioning, role assignment & password reset

**Status**: Done — see `PROJECT_PLAN.md` Section 11, "Admin dashboard: user provisioning, role assignment & password reset" (2026-07-12).

Four new workspace-scoped endpoints in `backend/src/routes/workspaces.js`, all gated on `requireWorkspaceAdmin`: `GET /:workspaceId/members` (roster — a gap the original design didn't spell out, found during implementation), `PATCH /:workspaceId/members/:userId` (role change with a last-admin guard), `POST /:workspaceId/users` (admin-provisioned account creation, transactional with the workspace-membership insert), `POST /:workspaceId/members/:userId/reset-password` (revokes the target's sessions, rejects targeting the caller's own id in favor of the self-service flow). Frontend: `UserManagementPanel.jsx`, reachable via a third "Manage Users" button in `WorkspaceSidebar`'s `adminToolsRow`. 155/155 backend tests, 13/13 e2e tests passing.

### Self-service password change

**Status**: Done — see `PROJECT_PLAN.md` Section 11, "Self-service password change" (2026-07-12).

`POST /api/auth/change-password` (`backend/src/routes/auth.js`), behind `requireAuth` and a new per-user `changePasswordLimiter`. `currentPassword` verified via `bcrypt.compare` (401 on mismatch), `newPassword` validated via the existing `assertValidPassword`, every outstanding refresh token revoked and a fresh access+refresh pair issued in the same response (current session keeps working, every other session is forced to re-authenticate), audited as `AUTH_PASSWORD_CHANGE`. Frontend: `ChangePasswordPanel.jsx` (same modal pattern as `AiSettingsPanel`/`AuditDashboard`), reachable from a "Change Password" control next to "Sign out" in `WorkspaceSidebar` for every user. 136/136 backend tests, 12/12 e2e tests passing.
