# Feature Requests — Running Plan

A living backlog of feature ideas for Silent Whisper. This is a planning document, not an implementation log — see `PROJECT_PLAN.md` Section 11 for what's actually been built.

## How this document is maintained

- Whenever a new feature is requested, a design is thought through (what changes, where, how it fits the existing authorization/audit/rate-limiting conventions in `PROJECT_PLAN.md`) and added below as its own entry, in the format under "Entry format."
- After each addition, every entry is re-ranked by utility — most useful first. Rank changes are expected over time as the app's shape changes; nothing here is permanent.
- An entry moves to **Done** (with a pointer to the `PROJECT_PLAN.md` Section 11 log entry that implemented it) once it ships, rather than being deleted — keeps the history of what was considered and why.
- Nothing in this document is authorization to build it. Design-only until explicitly requested.

## Entry format

```
### N. Title

**Status**: Proposed | Planned | Done
**Utility**: one line on why this matters / who it's for
**Origin**: how this came up (a request, a gap found during other work, etc.)

Design: what changes, where, how it's authorized/audited/tested, matching
existing PROJECT_PLAN.md conventions. Enough detail that implementation is
a matter of execution, not re-deciding the approach.
```

### 1. Enterprise authorization model: organizations, permission-based roles, invitations, no hard deletes

**Status**: Proposed
**Utility**: Very high. Fully supersedes and subsumes the previous version of this entry (two-tier system-admin/workspace-manager privilege). Today's model is a single flat `workspace_members.system_role` (`ADMIN`/`MEMBER`, `PROJECT_PLAN.md` Section 4) with nothing above the workspace: no organization concept, no explicit org membership, a conflated "admin" that means both "runs one workspace" and "manages system-wide audit/AI settings" (`requireAnyWorkspaceAdmin`, `authz/membershipService.js:45`), role checks that are raw string comparisons (`role !== 'ADMIN'`) scattered across `workspaces.js`/`ai.js`/`audit.js` and three places in the frontend, no invitation object, and no explicit guarantee that hard deletes never happen to anything the audit log references.
**Origin**: Requested directly as an explicit, fully-specified enterprise-grade authorization refactor — organizations as first-class objects, workspaces belonging to organizations, org membership explicit and separate from workspace membership, permission-set-based roles (not role strings), immutable user IDs with an active/disabled status, workspace status (active/archived) and visibility (private/discoverable) as first-class fields, a decoupled invitation object, a guaranteed at-least-one-owner-per-workspace invariant, status flips instead of hard deletes everywhere, and a migration that backfills a default organization for all existing data.

Design decisions locked in via direct clarification, since several readings of the request were genuinely ambiguous enough to derail the implementation if guessed wrong:
- **Self-service signup closes.** `POST /api/auth/signup` is removed entirely. Every account now originates from a system admin (direct creation) or from redeeming an invitation an org/workspace admin issued — a stricter reading than the previous version of this entry took (which explicitly preserved self-signup), because this request's own wording ("only system admins create accounts") was reconfirmed directly rather than inferred. This also retires every test helper that calls `/auth/signup` directly (`tests/helpers/testUsers.js`'s `signup()`, and the ad hoc `createWorkspace`/`addMember` helpers duplicated in `adminUserManagement.test.js`/`workspaceArchiving.test.js`) in favor of new shared seed helpers (see Tests below).
- **Manager archive is an owner-delegated toggle**, not a blanket grant or a permanent denial: `workspaces.managers_can_archive BOOLEAN NOT NULL DEFAULT false`, which the Owner controls. This is the literal reading of the request's "workspace manager can... optionally archive."
- **Visibility renames `PUBLIC` → `DISCOVERABLE`** (DB enum value, API field, frontend copy) to match the request's own vocabulary ("visibility private or discoverable"), touching the just-shipped self-service-subscription feature's enum, routes, and copy.
- **Invitation redemption is read as still permitting self-registration of credentials** (the invitee picks their own username/password when redeeming a valid, admin-issued token) — the alternative maximal reading ("no account ever created except by a system admin, full stop") would make "org and workspace admins can invite" a contradiction in terms, since an invite that still requires a system admin to finish creating the account is a notification, not delegated invite power. Flagged explicitly in case the stricter reading was actually intended.
- **Org creation is system-admin-only** and **no "every org needs ≥1 admin" invariant is enforced** (unlike the workspace-owner guarantee, which the request explicitly asks for) — the request only specifies the owner invariant at the workspace level, so nothing analogous is invented for orgs.
- **Removing someone from an org does not cascade-remove their workspace memberships** within that org — "workspace membership is explicit and separate from org membership" per the request, the opposite direction of the channel-membership cascade the previous version of this entry already established for workspace removal.

Design:
- **Users**: add `display_name VARCHAR(100) NOT NULL` (backfilled from `username`, then independently mutable) and `status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE'` (`ACTIVE`/`DISABLED`, enum enforced in `validation.js` like every other enum here — no DB `CHECK`, matching existing convention). `users.id` is already the immutable identifier; documented as such rather than changed.
- **Organizations (new)**: `organizations(id, name, created_at)` and `organization_members(organization_id, user_id, org_role, created_at)` with `org_role` = `ORG_ADMIN` | `ORG_MEMBER`, indexed on `user_id` matching `idx_workspace_members_user`'s existing precedent. `workspaces.organization_id UUID NOT NULL REFERENCES organizations(id)` (added nullable, backfilled by the migration below, then constrained).
- **Workspace status/visibility**: keep `archived_at`/`archived_by` as the status source of truth (`ACTIVE` when `archived_at IS NULL`) rather than adding a redundant `status` column — it already carries *when* and *by whom*, strictly more than a bare enum would. Add `managers_can_archive BOOLEAN NOT NULL DEFAULT false`. Rename `visibility` values `PUBLIC` → `DISCOVERABLE` (data update, not a column rename).
- **Workspace roles**: `workspace_members.system_role` moves from `ADMIN`/`MEMBER` to `OWNER`/`MANAGER`/`MEMBER`. Migration: existing `ADMIN` rows that aren't the workspace's `owner_id` become `MANAGER`; the row matching `workspaces.owner_id` becomes (or is inserted as, for the edge case `authz/membershipService.js:54` already flags — an owner with no `workspace_members` row at all) `OWNER` — guaranteeing "every workspace has ≥1 owner" holds immediately post-migration, not just going forward.
- **Invitations (new)**: `invitations(id, scope_type, organization_id, workspace_id, email, invited_role, invited_by, token_hash, status, expires_at, accepted_at, accepted_by, created_at)` — `token_hash` never stores the raw token, same convention as `refresh_tokens`; exactly one of `organization_id`/`workspace_id` is set per `scope_type`, enforced at the app layer. Direct add-by-username of an *already-existing* user (no token/email round trip) stays available on `POST /:workspaceId/members` and a new `POST /:orgId/members` — invitations are specifically for people who don't have an account yet, per the request's "decoupled from user creation."
- **Default-organization migration**: insert one `organizations` row ("Default Organization"), attach every existing workspace to it, then constrain `organization_id NOT NULL`. Every distinct user already in any `workspace_members` row gets an `organization_members` row as `ORG_MEMBER` in the default org (baseline access only — no invented promotion to `ORG_ADMIN` during migration, since nothing in the request asks for that).
- **No hard deletes**: `users`, `workspaces`, `organizations`, `channels`, `messages` already have zero delete routes today (confirmed by full route inventory) — so this is a defense-in-depth DB grant change, not a behavior change. `REVOKE DELETE ON users, workspaces, organizations, channels, messages FROM app_runtime_user`, mirroring the existing `audit_logs` append-only grant (`database/migrations/0007_grants.js`). `workspace_members`/`organization_members`/`channel_members`/`refresh_tokens`/`invitations` keep DELETE — removing a membership/session row is itself an audited event, not a hard-delete of an audited object.
- **Permission catalog (new, code not a DB table)**: `backend/src/authz/permissions.js` defines `PERMISSIONS` (e.g. `USERS_CREATE`, `USERS_DISABLE`, `USERS_VIEW_ALL`, `ORGS_VIEW_ALL`, `WORKSPACES_VIEW_ALL`, `AUDIT_VIEW`, `AI_SETTINGS_MANAGE`, `ORG_INVITE`, `ORG_MANAGE_MEMBERS`, `WORKSPACE_INVITE`, `WORKSPACE_MANAGE_MEMBERS`, `WORKSPACE_MANAGE_MANAGERS`, `WORKSPACE_ARCHIVE`, `WORKSPACE_CHANGE_VISIBILITY`, `WORKSPACE_TRANSFER_OWNERSHIP`) and static `ORG_ROLE_PERMISSIONS`/`WORKSPACE_ROLE_PERMISSIONS` maps (`OWNER` gets everything workspace-scoped; `MANAGER` gets invite/manage-members plus archive only if `managers_can_archive`; `MEMBER` gets nothing extra).
- **New authz primitives** (`authz/membershipService.js`): `hasWorkspacePermission`/`requireWorkspacePermission` and `hasOrgPermission`/`requireOrgPermission` replace every `role !== 'ADMIN'` check. Each checks, in order: (1) `users.is_system_admin` → full override, audited with `viaSystemAdminOverride: true` (the previous version of this entry's own design, generalized from "one boolean gate on two routes" to "one override inside every permission check"); (2) the caller's actual role via the permission maps above; (3) the `MANAGER`+`WORKSPACE_ARCHIVE` special case additionally requires `managers_can_archive`. Existence-hiding (404 for non-members) is preserved exactly as today.
- **Account lifecycle**: `scripts/create-first-admin.mjs` (new, offline CLI) creates the very first account directly with `is_system_admin = true` — necessary because signup is closed, so the first account must exist out-of-band; `scripts/grant-system-admin.mjs` (previous version's design) still promotes a later existing account. `POST /api/admin/users` (system-admin only) creates a bare account with no forced org/workspace attachment, retiring `POST /:workspaceId/users` entirely (account creation is no longer tied to administering some workspace). `POST /api/admin/users/:userId/disable` / `/enable` (system-admin only, global) — disabling revokes all refresh tokens and blocks future login/refresh (generic error, doesn't leak account existence/disabled-state); the outstanding ~15-minute access token is allowed to run out naturally rather than adding a DB check to every authenticated request, consistent with how short-lived access tokens are already the security boundary elsewhere.
- **Invitation routes**: `POST /api/organizations/:orgId/invitations` (`ORG_INVITE`) / `POST /api/workspaces/:workspaceId/invitations` (`WORKSPACE_INVITE`) create a token (raw value returned once — no email infra exists, same out-of-band precedent as admin-set passwords). `GET /api/invitations/:token` (public) shows context before redemption. `POST /api/invitations/:token/accept` (public) creates the account, attaches the invited-role membership, and marks the invitation `ACCEPTED`, all in one transaction, then logs the new user in immediately (same response shape as today's signup). `POST /api/invitations/:id/revoke` for the inviter or an equivalent permission-holder.
- **Workspace routes** (`workspaces.js`): `POST /` requires org membership (`organizationId` in body) and auto-assigns `OWNER`; `GET /discoverable` becomes org-scoped (`?organizationId=`, requires membership in that org — orgs as separate tenants, not one global discoverable pool); `archive`/`unarchive` collapse onto one `requireWorkspacePermission(WORKSPACE_ARCHIVE)` (removing today's owner-or-admin-vs-admin-only asymmetry, since the permission check itself now encodes owner-always/manager-if-delegated/system-admin-override); new `POST /:workspaceId/transfer-ownership` (`WORKSPACE_TRANSFER_OWNERSHIP`, owner only — target must already be a workspace member, old owner becomes `MANAGER`) and `POST /:workspaceId/visibility` (`WORKSPACE_CHANGE_VISIBILITY`, owner only); `PATCH /members/:userId` role changes gate on `WORKSPACE_MANAGE_MANAGERS` vs `WORKSPACE_MANAGE_MEMBERS` depending on target role, with a last-*owner* guard replacing today's last-*admin* guard; `DELETE /members/:userId` (the previous version's already-designed capability) is folded in here unchanged, including its `channel_members` cascade.
- **Organization routes (new)**: `backend/src/routes/organizations.js` — `POST /` (system-admin only), `GET /` (caller's orgs, or all if system admin), `GET`/`POST`/`PATCH`/`DELETE /:orgId/members` mirroring the workspace member-management shape, with the no-cascade and no-invariant scope boundaries noted above.
- **`audit.js`/`ai.js`**: gates move from `requireAnyWorkspaceAdmin` to `requireSystemPermission(AUDIT_VIEW)` / `requireSystemPermission(AI_SETTINGS_MANAGE)` — both are genuinely system-wide surfaces, tightened to actual system admins rather than "admin of any one workspace," same reasoning the previous version of this entry already used. `GET /workspaces/admin/all` (previous version's design) folds in here as `requireSystemPermission(WORKSPACES_VIEW_ALL)`.
- **Frontend** (high-level; not spec'd line-by-line here): an org-scoping layer above today's workspace list; every `ws.role === 'ADMIN'` check in `ChatShell.jsx:315`, `WorkspaceSidebar.jsx:408-409,468`, and `UserManagementPanel.jsx:254` replaced by a shared `hasPermission()` helper mirroring the backend catalog; a new public `/invite/:token` redemption page; a `SystemAdminPanel` replacing the current admin-tools bare-creation flow with org/workspace-agnostic account creation plus disable/enable controls; role `<select>`s updated to `OWNER`/`MANAGER`/`MEMBER` (owner not directly selectable — only via transfer-ownership).
- **Audit**: new action types `USER_STATUS_CHANGE`, `ORGANIZATION_CREATED`, `ORGANIZATION_MEMBERSHIP_CHANGE`, `WORKSPACE_OWNERSHIP_TRANSFERRED`, `WORKSPACE_VISIBILITY_CHANGED`, `INVITATION_CREATED`/`REVOKED`/`REDEEMED`, `SYSTEM_ADMIN_STATUS_CHANGE`, alongside the previous version's `viaSystemAdminOverride`/`ADMIN_ALL_WORKSPACES_VIEWED` design.
- **Tests**: every test file referencing `system_role`/`'ADMIN'`/`'MEMBER'` (full inventory: `authz/membershipService.js`, `validation.js`'s `WORKSPACE_ROLES`, `workspaces.js` at roughly a dozen sites, `ChatShell.jsx`, `WorkspaceSidebar.jsx`, `UserManagementPanel.jsx`, plus every test file's local `createWorkspace`/`addMember` helpers) needs updating to `OWNER`/`MANAGER`/`MEMBER` and org-scoped creation — the single largest mechanical cost of this refactor. New negative-authorization tests per permission: non-owner can't transfer ownership; manager without `managers_can_archive` can't archive; non-system-admin can't disable a user or create a bare account; removing the sole owner is blocked; transferring ownership to a non-member is blocked; disabled user can't log in and has refresh tokens revoked; expired/revoked/already-accepted invitation tokens are rejected; org-member-removal doesn't cascade workspace removal; a raw `DELETE FROM users` as `app_runtime_user` fails (verifies the new grant, mirroring however the existing `audit_logs` grant is currently tested). New shared `tests/helpers/` additions (`seedOrg`, `seedInvitation`, `seedSystemAdmin`) replace the signup-based and duplicated per-file seed helpers this entry retires.

**Implementation note**: given the size here (schema, permission model, invitations, ~8 backend route files, a breaking role rename, and several frontend admin panels), this is expected to be sequenced into separate efforts when built — schema + permission model + authz module first, then invitation/org routes, then frontend, then the full test-suite migration — rather than landing as one commit.

Ships the schema, the permission catalog, the `membershipService.js` rewrite, and the full cutover of every existing route/frontend call site to the new `OWNER`/`MANAGER`/`MEMBER` model — scoped tighter than this entry's original Design in a few places (a slice-1-only `WORKSPACE_ROLE_PERMISSIONS` map covering just what today's routes need, `requireSystemPermission`'s OR-fallback kept rather than tightened to `is_system_admin`-only, `managers_can_archive` added to the schema but not yet enforced), all flagged inline in the Section 11 entry as deliberate, temporary narrowings — not a re-litigation of the design above. ### 2. Cross-channel "Catch Me Up" workspace digests

**Status**: Proposed
**Utility**: High. This is a natural next step for the existing local AI features: it turns unread mentions and important channel activity into a short operational brief for someone returning from a multi-day break, instead of forcing them to manually scan every backlog thread.
**Origin**: Requested directly as part of "Advanced Silent Whisper AI Enhancements," with an explicit `POST /api/ai/workspace-digest` endpoint, variable time window, context-window batching, and SSE/WebSocket streaming constraints.

Design:
- **API**: add `POST /api/ai/workspace-digest`, behind `requireAuth`, with a body such as `{ workspaceId, sinceHours?, sinceDays?, starredChannelIds?, includeMentionsOnly? }`. Clamp the requested window to a configured maximum so a single request cannot pull years of message history into an LLM job.
- **Source selection**: query unread messages that mention the user plus unread threads/messages in the user's starred channels within the requested time window. If starred channels do not exist yet, this feature either depends on adding a small `starred_channels(user_id, channel_id, created_at)` table first or starts with an explicit channel list and leaves first-class starring as a prerequisite companion feature.
- **Authorization**: every selected message must come from a channel the caller can currently read. Private channels, DMs, and group DMs follow the same membership checks as message history and AI summarize/extract endpoints; no digest can leak messages from channels the caller cannot access.
- **Unread model**: this feature needs a durable read-state source if one is not already present. Add or depend on per-user/channel read markers (`last_read_message_id` or `last_read_at`) so "unread" is not guessed from timestamps alone. Direct mentions can be detected either from the future mention feature's persisted data if it exists, or by a bounded username-token query over recent messages as a v1 fallback.
- **Prompt batching**: batch and slice message text to fit the configured local model's context window (`mistral`, vLLM-hosted models, etc.). Preserve enough metadata in each chunk (channel, thread root, author, timestamp) for the final summary to cite where action items came from. Summaries should be hierarchical: per-batch summaries first, then a final merge pass for a unified digest.
- **Streaming**: stream markdown back to the client using SSE or WebSockets. SSE is likely the simplest REST-shaped fit (`text/event-stream`) unless the existing WebSocket client already has a robust request/response pattern for AI streams. The UI should render partial markdown as it arrives and allow cancellation without leaving the provider request running indefinitely.
- **Output shape**: generate a concise markdown digest with sections for urgent mentions, action items assigned to or involving the user, unresolved questions, decisions made while away, and links/back-references to the originating channel/thread where possible. Treat all model output as untrusted text when rendering, consistent with the existing AI output rules.
- **Rate limiting/concurrency**: enforce the same global `LLM_MAX_CONCURRENT_REQUESTS` style cap used by channel summaries, plus a stricter per-user digest rate limit because this endpoint can scan many channels and run multiple prompt batches.
- **Audit**: audit as an AI operation with workspace id, requested window, selected message count, chunk count, provider/model, prompt version, and truncated input length. Do not log raw message content in the audit payload.
- **Tests**: endpoint selects only authorized unread mentions/starred-channel messages; time-window clamping works; long inputs are chunked below configured limits; provider streams are forwarded incrementally to the client; cancellation closes the upstream provider request; audit rows record metadata without raw content.

### 3. Default workspace on login

**Status**: Proposed
**Utility**: Medium. Today, `ChatShell.jsx`'s initial-load effect (`frontend/src/components/ChatShell.jsx:174-177`) auto-selects `ws[0].id` — whichever workspace `GET /workspaces` happens to return first (`ORDER BY created_at ASC`, i.e. oldest-joined), not a deliberate choice. For a user in exactly one workspace this is already a no-op (there's only one candidate), but for anyone in two or more it's arbitrary — the main window can land on a stale or rarely-used workspace, leaving the message pane empty until the user manually clicks the one they actually wanted. Letting them pin a specific workspace fixes that dead-space-on-startup gap directly.
**Origin**: Requested directly, framed around minimizing empty main-window space on startup. Interpreted as an **explicit, persistent, user-chosen default** (a deliberate pin, surviving across logins/devices) — distinct from a simpler "just remember whatever I looked at last" session-based approach, which the request's wording ("select a default workspace") doesn't actually ask for; flagging the distinction the same way past entries have flagged their own interpretation calls.

Design:
- **Schema**: add `default_workspace_id UUID NULL REFERENCES workspaces(id) ON DELETE SET NULL` to `users` (new migration — `0010` is already taken by the shipped workspace-visibility migration, so this is `0011_default_workspace.js`). Nullable, defaulting to `NULL` for every existing and newly-created user, preserving today's `ws[0]` fallback behavior for anyone who never sets one. `ON DELETE SET NULL` rather than a hard block: there's no workspace-delete endpoint today (only archive/unarchive), so this is future-proofing, not a live concern, but a stale FK silently blocking an unrelated future delete would be a surprising failure mode for a feature this minor.
- **Setting it**: `PUT /api/auth/me/default-workspace`, behind `requireAuth`, body `{ workspaceId: <uuid> | null }` (`null` clears it, reverting to the `ws[0]` fallback). Validates the target via `requireWorkspaceMember` (`authz/membershipService.js`) before storing it — same existence-hiding 404 for a non-member/nonexistent workspace as every other membership-gated route, so this can't be used to confirm a private workspace's existence or pin a workspace the caller doesn't actually belong to. No admin/role requirement beyond plain membership — this is a personal preference about the caller's own view, not a workspace-administration action, so any member (not just `ADMIN`) can set it for themselves. Returns `{ user }` (the same `{ id, username, email, defaultWorkspaceId }` shape below), mirroring `POST /auth/change-password`'s existing "return the updated user/session state" pattern.
- **Surfacing it**: add `default_workspace_id` (as `defaultWorkspaceId`) to the user object returned by `GET /api/auth/me`, `POST /api/auth/login`, and `POST /api/auth/signup` (`backend/src/routes/auth.js`) — the same three places that already return `{ id, username, email }`. This means `AuthContext.jsx`'s existing `user` state carries it through session-restore-on-reload for free, with no new fetch needed.
- **Not audited**: unlike membership/role changes, this doesn't affect authorization or any other user, and doesn't fit the audit log's existing scope (security-relevant events — failed logins, membership/role changes, admin actions, AI operations per `PROJECT_PLAN.md` Section 4's Forensic Security Audit Log). Treated the same as any other cosmetic personal preference — not every state change in this app is an audit event, and this one isn't security-relevant enough to be one.
- **Not rate-limited**: a personal single-value preference update, same class of "not attacker-interesting, not separately limited" as workspace/channel creation.
- **Frontend — control**: `WorkspaceSidebar.jsx`'s existing per-workspace "•••" overflow menu (`workspaceMenuItems`, `frontend/src/components/WorkspaceSidebar.jsx:410-413`, which already holds Invite/Archive) gains one more entry — a `checked`-style item (reusing `Menu.jsx`'s existing `menuitemcheckbox` rendering, the same mechanism the Light/Dark toggle entry already uses) labeled "Default workspace", `checked: ws.id === user.defaultWorkspaceId`. Selecting it when unchecked sets that workspace as the default; selecting it when already checked clears it (sends `null`) — a toggle, not a one-way pin, available to every member regardless of role (unlike Invite/Archive, which stay role-gated). `AuthContext.jsx` gains a `setDefaultWorkspace(workspaceId)` method (mirroring how `changePassword` already wraps its own API call and updates `user` state) so the sidebar's checkmark and any other `user.defaultWorkspaceId` read update immediately, without a page reload.
- **Frontend — applying it on load**: `ChatShell.jsx`'s initial-load effect (line 174-177 today) changes from unconditionally picking `ws[0].id` to: if `user.defaultWorkspaceId` is set *and* present in the just-fetched `ws` list (defends against a default pointing at a workspace the user has since lost membership in — no separate cleanup job needed, this check is cheap and already has the list in hand), select it; otherwise fall back to `ws[0].id` exactly as today.
- **Tests**: setting a default on a workspace the caller belongs to succeeds and is reflected in `GET /api/auth/me`/login/signup responses; setting it on a workspace the caller isn't a member of 404s (existence-hiding, not 403); setting `null` clears it; a plain `MEMBER` (not just `ADMIN`) can set their own default; the frontend's initial-selection logic prefers a valid `defaultWorkspaceId` over `ws[0]` when both are present, and falls back to `ws[0]` when the stored default is no longer in the caller's workspace list (e.g. removed from that workspace since setting it).
